# 멀티 에이전트 토론과 협업

> Du et al.(ICML 2024, "Society of Minds")은 독립적으로 답을 제안한 뒤 R 라운드에 걸쳐 서로를 반복적으로 비평하여 수렴하는 N개의 모델(model) 인스턴스를 실행한다. 사실성, 규칙 준수, 추론을 개선한다. 희소(sparse) 토폴로지가 토큰 비용에서 완전 메시(full mesh)를 이긴다.

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 14 · 12 (Workflow Patterns), Phase 14 · 05 (Self-Refine and CRITIC)
**Time:** ~60분

## 학습 목표 (Learning Objectives)

- 토론 프로토콜을 설명하기: N명의 제안자(proposer), R 라운드, 공유 답으로의 수렴(converge).
- 토론이 왜 사실성, 규칙 준수, 추론을 개선하는지 기술하기.
- 희소 토폴로지를 설명하기: 모든 토론자가 다른 모두를 볼 필요는 없다.
- 완전 메시와 희소 변형을 갖춘 스크립트화된 LLM에 대한 stdlib 토론을 구현하고; 토큰 비용 대 정확도를 측정하기.

## 문제 (The Problem)

Self-Refine(Lesson 05)는 하나의 모델이 자신을 비평하는 것이다 — 집단사고(groupthink)의 위험이 있다. CRITIC(Lesson 05)은 비평을 외부 도구에 그라운딩한다 — 항상 가용한 것은 아니다. 토론은 세 번째 모드를 도입한다: 다수의 인스턴스, 교차 비평, 의견 불일치에 의한 수렴.

## 개념 (The Concept)

### Society of Minds (Du et al., ICML 2024)

- N개의 모델 인스턴스가 같은 질문에 대해 독립적으로 답을 제안한다.
- R 라운드에 걸쳐, 각 모델은 다른 모델의 제안을 읽고 비평한다.
- 모델은 비평에 기반하여 답을 갱신한다.
- R 라운드 후, 수렴된 답을 반환한다.

원래 실험은 비용 때문에 N=3, R=2를 사용했다. 어려운 문제(MMLU, GSM8K, Chess Move Validity, 전기 생성)에서는 에이전트가 더 많고 라운드가 더 많을수록 정확도가 개선된다.

교차 모델 조합이 단일 모델 토론을 이긴다: ChatGPT + Bard 함께 > 어느 하나 단독.

### 희소 토폴로지

"Improving Multi-Agent Debate with Sparse Communication Topology"(arXiv:2406.11776, 2024-2025)는 완전 메시 토론이 항상 최적은 아님을 보여주었다. 희소 토폴로지(별, 링, 허브 앤 스포크)는 더 낮은 토큰 비용으로 정확도를 맞출 수 있다. 각 토론자는 동료의 부분집합만 본다.

함의:

- 완전 메시 N=5, R=3 = 5 × 3 = 제안 15개, 각각 4명의 동료를 읽음 = 비평 연산 60회.
- 별 N=5, R=3 (허브 1개 + 스포크 4개) = 제안 15개, 스포크는 허브만 읽음 = 비평 연산 12회.

### 토론이 도움이 될 때

- **사실성.** N개의 독립 제안, 교차 검사가 환각을 줄인다.
- **규칙 준수.** 체스 수의 유효성 — 한 모델이 규칙을 놓치면 다른 모델이 잡아낸다.
- **개방형 추론.** 다수의 틀이 올바른 답으로 좁혀 들어간다.

### 토론이 해로울 때

- **지연 시간에 민감한 UX.** N × R의 직렬 라운드는 당신에게 없을 수도 있는 지연 시간이다.
- **비용에 민감한 규모.** 질문당 N × R 토큰.
- **단순한 사실 조회.** 한 번의 조회가 다섯 번의 토론보다 싸다.

### 2026년 실용적 구현체

- **Anthropic 오케스트레이터-워커(orchestrator-workers)**(Lesson 12) — 종합(synthesis) 단계를 갖춘 토론의 한 변형.
- **LangGraph 슈퍼바이저(supervisor)**(Lesson 13) — 중앙 라우터 + 전문가 에이전트가 토론을 노드로 구현할 수 있다.
- **OpenAI Agents SDK**(Lesson 16) — 에이전트가 반복적 비평을 위해 서로 핸드오프(handoff)한다.
- **멀티 에이전트 평가** — 평가 신호를 위해 토론 + 평가자-최적화자(evaluator-optimizer)를 짝지음.

### 이 패턴이 잘못되는 지점

- **수렴 붕괴.** 모든 에이전트가 첫 번째 틀린 답으로 수렴한다. 필수 불일치 라운드로 완화하라.
- **허브 실패.** 별 토폴로지에서 나쁜 허브가 모두를 오염시킨다. 순환시키거나 다수의 허브를 사용하라.
- **프롬프트 동질화.** 모든 에이전트가 같은 프롬프트를 사용한다; 같은 답을 낸다. 다양한 프롬프트 및/또는 모델을 사용하라.

## 직접 만들기 (Build It)

`code/main.py`는 stdlib 토론을 구현한다:

- `Debater` 클래스(토론자별 의견 드리프트를 갖춘 스크립트화된 LLM).
- `FullMeshDebate`와 `SparseDebate` 러너.
- 세 가지 질문: 사실 하나, 규칙 기반 하나, 추론 하나.
- 지표: 수렴된 답, 수렴까지의 라운드, 총 비평 연산.

실행:

```
python3 code/main.py
```

출력: 프로토콜별 정확도와 비용; 희소가 더 낮은 비용으로 3개 중 2개 질문에서 완전 메시를 맞춘다.

## 라이브러리로 써보기 (Use It)

- 단순한 2~3명 워커 토론에는 **Anthropic 오케스트레이터-워커**.
- 체크포인팅(checkpointing)을 갖춘 상태 유지 멀티 라운드 토론에는 **LangGraph**.
- 연구나 전문화된 정확성 보장에는 **맞춤(custom)**.

## 산출물 (Ship It)

`outputs/skill-debate.md`는 설정 가능한 토폴로지, N, R, 그리고 수렴 규칙을 갖춘 멀티 에이전트 토론을 스캐폴딩한다.

## 연습 문제 (Exercises)

1. "강제 불일치" 규칙을 구현하라: 라운드 1에서 모든 토론자가 서로 다른 제안을 내야 한다. 수렴 속도에 미치는 영향을 측정하라.
2. 신뢰도 가중 집계를 추가하라: 토론자가 (답, 신뢰도)를 반환하고; 집계자가 신뢰도로 가중한다. 도움이 되는가?
3. 한 "에이전트"를 다른 의견을 가진 다른 스크립트화된 LLM으로 교체하라. 이질성이 정확도를 개선하는가?
4. 당신의 3개 질문에서 완전 메시 대 희소의 토큰 비용을 측정하라. 비용 대 정확도를 그려라.
5. Society of Minds 논문을 읽어라. 당신의 장난감을 N=5, R=3으로 이식하라. 무엇이 깨지는가? 무엇이 좋아지는가?

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| Debate | "멀티 에이전트 비평" | N명 제안자, R 라운드의 교차 비평, 수렴 |
| Full mesh | "모두가 모두를 읽음" | 모든 토론자가 매 라운드 모든 동료를 읽음 |
| Sparse topology | "제한된 동료 시야" | 토론자가 동료의 부분집합만 읽음 |
| Hub-and-spoke | "별 토폴로지" | 중앙 토론자 1명, N-1개 스포크가 허브만 읽음 |
| Convergence | "합의" | 토론자가 공유 답으로 수렴 |
| Society of Minds | "Du et al. 토론 논문" | ICML 2024 멀티 에이전트 토론 방법 |
