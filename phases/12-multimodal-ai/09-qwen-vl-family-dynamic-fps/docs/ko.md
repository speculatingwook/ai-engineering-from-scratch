# Qwen-VL 계열과 동적 FPS 비디오(Qwen-VL Family and Dynamic-FPS Video)

> Qwen-VL 계열 — Qwen-VL(2023), Qwen2-VL(2024), Qwen2.5-VL(2025), Qwen3-VL(2025) — 은 2026년 가장 영향력 있는 오픈 비전-언어 모델 혈통이다. 각 세대는 나머지 오픈 생태계가 12개월 안에 따라 한 결정적 아키텍처 선택을 하나씩 했다: M-RoPE로 구현한 네이티브 동적 해상도(dynamic resolution), 절대 시간 정렬을 갖춘 동적 FPS 샘플링(sampling), ViT 내 윈도우 어텐션(window attention), 구조화된 에이전트(agent) 출력 포맷이다. Qwen3-VL에 이르러 레시피는 안정화되었다. 네이티브 종횡비 입력을 받는 2D-RoPE-ViT 인코더(encoder), 큰 Qwen3 언어 베이스로 이어지는 MLP 투영기(projector), OCR·그라운딩(grounding)·에이전트 행동을 일급 목표로 강조한 학습 단계로 이뤄진다. 이 레슨은 모든 손잡이가 왜 그 자리에 있는지 이해하도록 그 계열을 연대순으로 읽는다.

**Type:** Learn
**Languages:** Python (stdlib, M-RoPE encoder + dynamic-FPS sampler)
**Prerequisites:** Phase 12 · 06 (patch-n'-pack)
**Time:** ~120분

## 학습 목표 (Learning Objectives)

- M-RoPE의 3축 회전(시간, 높이, 너비)을 계산하고 세 가지 모두가 필요한 이유 설명하기.
- 비디오에 대한 동적 FPS 샘플링 전략을 고르고 초당 토큰(tokens-per-second) 대 이벤트 탐지 정확도를 따져 보기.
- 네 가지 Qwen-VL 세대 업그레이드를 순서대로 이름 대고 각각이 무엇을 가능하게 했는지 말하기.
- Qwen2.5-VL 스타일 JSON 에이전트 출력 포맷을 연결하고 VLM 응답에서 구조화된 도구 호출(tool call) 파싱하기.

## 문제 (The Problem)

Qwen-VL은 LLaVA-1.5와 BLIP-2에 대한 직접적 응답으로 2023년 8월에 출시되었다. Qwen 팀이 겨냥한 간극은 세 가지였다: 해상도, 비디오, 구조화된 출력.

해상도: LLaVA-1.5는 336x336에서 돌아갔다. 사진에는 괜찮지만, 중국어 송장이나 빽빽한 스프레드시트 스크린샷에는 쓸모없다. Qwen-VL의 첫 혁신은 448x448과 그라운딩된 바운딩 박스(bounding-box) 출력이었고, 덕분에 모델이 사물을 가리킬 수 있었다.

비디오: Video-LLaMA는 프레임별 인코더를 쌓아 LLM에 입력했다. 짧은 클립에서는 잘 돌아갔지만, 시간 축이 곧 신호인 수 분짜리 비디오에서는 그렇지 못했다. Qwen 팀은 시간을 이해하는 단일 인코더를 원했다.

구조화된 출력: LLaVA는 자유형식 텍스트를 내보냈다. 에이전트는 JSON이 필요하다. Qwen-VL은 텍스트로서의 바운딩 박스 좌표를 포함한 명시적 JSON 출력 포맷으로 학습했다.

모든 Qwen-VL 세대는 이 세 축 중 하나를 확장한다.

## 개념 (The Concept)

### Qwen-VL (2023년 8월)

첫 세대: 인코더로 OpenCLIP ViT-bigG/14(25억 파라미터(parameter)), LLama 호환 Q-Former(256개 쿼리(query)로 1단계), Qwen-7B 베이스. 기여:

- 448x448 해상도(당시 오픈 VLM의 최첨단(SOTA)).
- 그라운딩: 명시적 좌표 토큰 출력을 갖춘 이미지-텍스트 쌍으로 학습. "The cat is at <box>(112, 204), (280, 344)</box>".
- 처음부터 중국어 + 영어 다국어 학습.

당시 벤치마크(benchmark): 영어에서 GPT-4V와 경쟁력이 있고, 중국어에서 지배적. 그라운딩 지도 신호가 진짜 헤드라인이었다.

### Qwen2-VL (2024년 9월) — M-RoPE와 네이티브 해상도

Qwen2-VL은 고정 해상도 + Q-Former 스택을 네이티브 동적 해상도 ViT 인코더로 대체했다. 핵심 변화:

- 네이티브 동적 해상도. ViT는 28로 나누어지는 임의의 HxW를 받아들인다(2배 공간 병합(spatial merge)을 쓴 패치 14). 1120x672 이미지(40x24 병합 패치)는 960개 시각 토큰(visual token)을 만든다. 크기 조정 없음, 타일링 없음, 썸네일 없음.
- M-RoPE(멀티모달 RoPE). 각 토큰이 1D 대신 3D 위치(t, h, w)를 지닌다. 이미지에는 t=0, 비디오에는 t = frame_index. RoPE가 축별 주파수로 쿼리/키(key) 벡터를 회전시킨다. 위치 임베딩 테이블 없음.
- MLP 투영기. Q-Former를 버린다. 병합된 패치 토큰에 2층 MLP를 쓴다.
- 동적 FPS 비디오. 비디오는 기본적으로 1-2 FPS로 샘플링되지만, 모델은 임의의 프레임 개수를 받아들인다.

결과: Qwen2-VL-7B는 여러 멀티모달 벤치마크에서 GPT-4o와 동등했고 DocVQA에서는 앞섰다(94.5 대 88.4). 아키텍처 변경이 결정적 한 수였다.

### Qwen2.5-VL (2025년 2월) — 동적 FPS + 절대 시간

Qwen2.5-VL의 큰 전환은 비디오였다. 동적 FPS는 단지 "필요할 때 더 많은 프레임을 샘플링"하는 것이 아니다. 논문은 다음을 형식화했다:

- 절대 시간 토큰. 위치 인덱스(프레임 0, 1, 2...) 대신 실제 타임스탬프를 쓴다. "0:04에 고양이가 점프한다." 모델은 프레임 토큰 사이에 인터리브된(interleaved) `<time>0.04</time>` 토큰을 본다.
- 동적 FPS. 느린 영상에는 1 FPS, 액션에는 4+ FPS로 샘플링한다. 사용자나 트레이너가 고른다. M-RoPE가 적응한다.
- ViT 내 윈도우 어텐션. 공간 어텐션은 처리량(throughput)을 위해 윈도우화된다(블록 내 국소적). 몇 개 층마다 전역 어텐션.
- 명시적 JSON 출력 포맷. 도구 호출 데이터로 학습: "{\"tool\": \"click\", \"coords\": [380, 220]}". 곧바로 에이전트에 쓸 수 있다.
- MRoPE-v2 스케일링. 위치가 최대 입력 크기에 따라 스케일링되어, 10분짜리 비디오가 주파수 범위를 소진하지 않는다.

벤치마크: Qwen2.5-VL-72B는 대부분의 비디오 벤치마크에서 GPT-4o를 이기고, 문서에서 Gemini 2.0과 동등하며, GUI 그라운딩에서 오픈 모델 최첨단을 세운다(ScreenSpot: 정확도 84% 대 GPT-4o의 38%).

### Qwen3-VL (2025년 11월)

Qwen3-VL은 재발명이 아니라 기존 성과를 다지는 점진적 업그레이드다. 더 큰 LLM 백본(Qwen3-72B), 확장된 학습 데이터, 개선된 OCR, Qwen3 "사고 모드(thinking mode)"로 강화한 추론이 핵심이다. ViT와 M-RoPE는 그대로다. 논문은 아키텍처보다 데이터와 학습 개선에 초점을 맞춘다.

혈통의 교훈: 2025년 무렵 Qwen-VL 아키텍처는 안정화되었다. 추가 세대는 원시 단위(primitive)가 아니라 연산과 데이터를 스케일링한다.

### M-RoPE 수학적으로 (M-RoPE mathematically)

고전적 RoPE는 차원 `d`의 쿼리 `q`를 짝지어진 좌표를 써서 위치 `m`만큼 회전시킨다:

```
q_rot[2i]   = q[2i]   * cos(m * theta_i) - q[2i+1] * sin(m * theta_i)
q_rot[2i+1] = q[2i]   * sin(m * theta_i) + q[2i+1] * cos(m * theta_i)
theta_i     = 10000^(-2i/d)
```

M-RoPE는 은닉 차원을 세 개의 대역(band)으로 나눈다. `d = 96`이라 하자. 32차원을 시간에, 32를 높이에, 32를 너비에 할당한다. 각 대역은 자신의 축 위치만큼 회전한다. (t=5, h=10, w=20)에 있는 패치는 세 대역에 적용되는 회전 `R_t(5)`, `R_h(10)`, `R_w(20)`을 얻는다.

텍스트 토큰은 `t = text_index, h = 0, w = 0`(또는 정규화된 선택)을 써서 호환성을 유지한다. 비디오 프레임은 `t = frame_time, h = row, w = col`을 쓴다. 단일 이미지는 `t = 0`을 쓴다.

이점: 위치 인코딩 하나가 분기 코드나 별도의 위치 테이블 없이 텍스트, 이미지, 비디오를 처리한다.

### 동적 FPS 샘플링 로직 (Dynamic-FPS sampling logic)

지속 시간 `T`초의 비디오와 목표 토큰 예산 `B`가 주어지면:

1. 감당할 수 있는 최대 FPS를 계산한다: `fps_max = B / (T * tokens_per_frame)`.
2. `fps <= fps_max`를 만족하는 목표 FPS를 `{1, 2, 4, 8}`에서 고른다.
3. 움직임이 크면(옵티컬 플로 휴리스틱 또는 명시적 사용자 요청), 더 높은 FPS를 고른다. 움직임이 작으면, 더 낮게 고른다.
4. 선택한 FPS로 균등하게 샘플링하고, 프레임 사이에 `<time>t</time>` 토큰을 삽입한다.

Qwen2.5-VL은 이 로직을 암시적으로 학습한다. 추론 시 사용자가 `fps` 파라미터로 제어한다. 프레임당 81토큰인 4 FPS의 60초짜리 액션 시퀀스 = 19440토큰으로, 32k 컨텍스트(context)에서 다룰 만하다.

### 구조화된 에이전트 출력 (Structured agent output)

Qwen2.5-VL의 에이전트 학습은 명시적으로 구조화된 도구 호출을 겨냥한다:

```
{
  "tool": "mouse_click",
  "coords": [1024, 512],
  "button": "left",
  "modifier": null
}
```

파싱은 결정론적이다. 모델 출력에 JSON.parse를 돌리면 된다. 정규식과 모호성 처리가 필요했던 자유형식 "click at (1024, 512)"와 비교해 보라. 이 전환 덕분에 Qwen2.5-VL의 ScreenSpot 점수가 Qwen2-VL의 55%에서 84%로 뛰었다.

## 라이브러리로 써보기 (Use It)

`code/main.py`는 다음을 구현한다:

- 텍스트, 이미지 패치, 비디오 프레임을 섞은 패킹된 시퀀스에 대한 M-RoPE 위치 계산.
- 동적 FPS 샘플러: (지속 시간, 예산, motion_level)이 주어지면, FPS를 고르고 프레임 타임스탬프를 내보낸다.
- 좌표 필드를 가진 도구 호출 응답을 처리하는 장난감 Qwen2.5-VL JSON 출력 파서.

실행하고, 5분짜리 비디오에서 고정 FPS를 동적 FPS로 바꿀 때의 차이를 체감하라.

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|-----------------|------------------------|
| M-RoPE | "멀티모달 RoPE" | 은닉 차원에 시간, 높이, 너비 대역을 가진 3D 회전 위치 임베딩 |
| 동적 FPS(Dynamic FPS) | "스마트 샘플링" | 움직임, 지속 시간, 토큰 예산에 따라 비디오별로 선택되는 프레임 샘플링 속도 |
| 절대 시간 토큰(Absolute time token) | "타임스탬프 토큰" | 모델이 프레임 인덱스가 아니라 실제 초를 보도록 시퀀스에 인터리브된 `<time>t</time>` |
| 윈도우 어텐션(Window attention) | "국소 어텐션" | 속도를 위해 작은 윈도우로 제한된 공간 셀프 어텐션. 주기적으로 전역 어텐션이 추가된다 |
| 구조화된 에이전트 출력(Structured agent output) | "JSON 모드" | 좌표와 도구 이름을 가진 파싱 가능한 JSON을 내보내도록 VLM을 가르치는 학습 데이터 지도 신호 |
| min_pixels / max_pixels | "해상도 경계" | 총 픽셀 개수, 따라서 토큰 개수를 상한 짓는 요청별 Qwen2.5-VL 제어 |
| 그라운딩(Grounding) | "그것을 가리키기" | 바운딩 박스 좌표를 텍스트 토큰으로 출력하는 것. Qwen-VL v1부터 쓰임 |
