# LLaVA-OneVision: 단일 이미지, 멀티 이미지, 비디오를 하나의 모델에(LLaVA-OneVision: Single-Image, Multi-Image, Video in One Model)

> LLaVA-OneVision(Li et al., 2024년 8월) 이전의 오픈 VLM 세계는 혈통이 서로 갈라져 있었다. 단일 이미지를 위한 LLaVA-1.5, Mantis와 VILA 같은 멀티 이미지 모델, Video-LLaVA와 Video-LLaMA 같은 비디오 모델이 따로 있었다. 각 모델은 자기 벤치마크(benchmark)에서는 이겼지만 나머지에서는 졌다. LLaVA-OneVision은 하나의 커리큘럼으로 세 시나리오를 모두 지배하는 단일 모델을 학습시킬 수 있고, 창발적(emergent) 작업 전이 효과(단일 이미지 기술이 비디오로 옮겨 가고, 멀티 이미지 추론이 단일 이미지로 옮겨 감)가 전문가들의 합을 이긴다고 주장했다. 그 레시피는 의심스러울 만큼 단순하다. 시나리오 전반에 일정하게 유지되는 시각 토큰(visual token) 예산, 그리고 단일 이미지에서 OneVision(멀티 이미지)으로, 다시 비디오로 옮겨 가는 명시적 커리큘럼이다. 이 레슨은 그 예산, 커리큘럼, 창발적 행동을 읽는다.

**Type:** Build
**Languages:** Python (stdlib, token budget solver + curriculum planner)
**Prerequisites:** Phase 12 · 05 (LLaVA), Phase 12 · 06 (any-resolution)
**Time:** ~180분

## 학습 목표 (Learning Objectives)

- 단일 이미지, 멀티 이미지, 비디오 입력 전반에 일정하게 유지되는 시각 토큰 예산 설계하기.
- 파국적 망각(catastrophic forgetting) 없이 단일 이미지에서 비디오로 기술을 전이하는 학습 커리큘럼 순서 정하기.
- 커리큘럼을 제대로 했을 때 같은 파라미터(parameter) 개수에서 단일 모델이 전문가들을 이기는 이유 설명하기.
- LLaVA-OneVision이 보고한 세 가지 창발 능력 이름 대기: 멀티 카메라 추론, 마크 집합(set-of-mark) 프롬프팅, 아이폰 스크린샷 에이전트(agent).

## 문제 (The Problem)

이미지, 멀티 이미지, 비디오는 각각 모델을 다르게 압박한다.

단일 이미지는 OCR과 미세한 세부를 잡으려고 고해상도 토큰(AnyRes, 약 2880개 시각 토큰)을 원한다. 샘플당 예산: 이미지 하나, 2880토큰.

멀티 이미지는 이미지를 가로지르는 추론이 컨텍스트(context)에 맞도록 중간 해상도(각 약 576토큰)의 여러 이미지를 원한다. 샘플당 예산: 4-8개 이미지, 각 576, 2300-4600토큰.

비디오는 시간적 동역학을 잡으려고 저해상도(풀링(pooling) 후 프레임당 약 196토큰)의 많은 프레임을 원한다. 샘플당 예산: 8-32개 프레임, 각 196, 1600-6200토큰.

별개의 모델을 학습시키면 예산을 하나만 고르면 된다. 하나의 모델을 학습시키면, 컨텍스트를 넘치게 하지 않으면서 시나리오 전반에 예산이 합리적으로 스케일링되어야 한다.

OneVision 이전의 기본 답은 "하나의 시나리오를 학습하고 나머지는 무시하라"였다. Video-LLaVA는 추가 학습 단계로 이미지 모델에 비디오를 개조해 붙였다. LLaVA-NeXT는 타일링(tiling)으로 멀티 이미지 지원을 추가했다. 어느 쪽도 세 가지 모두를 깔끔하게 처리하지 못했다.

## 개념 (The Concept)

### OneVision 토큰 예산 (The OneVision token budget)

LLaVA-OneVision은 샘플당 약 3000-4000토큰의 통합 시각 토큰 예산을 고르고, 이를 시나리오별로 다르게 할당한다.

- 단일 이미지: AnyRes-9(3x3 타일 + 썸네일), 각 타일은 729개 패치(patch)의 384, 공격적 쌍선형(bilinear) 풀링 2x2 → 타일당 182. 총합: 9 * 182 + 182 = 1820토큰. 또는 타일당 729인 AnyRes-4 = 2916 + 729.
- 멀티 이미지: 각 이미지는 중간 해상도(384, 타일링 없음), 풀링 없이 729토큰. 6개 이미지 예산 → 4374토큰.
- 비디오: 공격적 3x3 쌍선형 풀로 384 해상도의 32프레임 → 프레임당 81토큰. 총합: 32 * 81 = 2592토큰.

이 할당은 총 토큰을 대략 일정하게 유지한다. LLM은 자신의 컨텍스트를 넘치게 하는 배치(batch)를 결코 보지 않는다. 인코더(encoder)는 시나리오별로 다른 기하를 만들지만, LLM은 같은 예산을 소비한다.

### 3단계 커리큘럼 (The three-stage curriculum)

LLaVA-OneVision은 세 단계로 학습한다.

1. 단일 이미지 SFT(stage SI). 모든 데이터가 단일 이미지 + 텍스트다. 고해상도 AnyRes 입력으로 학습한다. 이 단계가 지각, OCR, 세밀한 이해를 가르친다. LLaVA-NeXT 데이터에 OneVision 전용 단일 이미지 데이터를 더해 쓴다.
2. OneVision SFT(stage OV). 단일 이미지 + 멀티 이미지 + 비디오(균등 샘플링된 프레임)를 섞는다. 통합 토큰 예산으로 학습한다. 이 단계가 모델에게 이질적 배치 형상을 처리하는 법을 가르친다. 가중치(weight) 재설정은 없으며, stage SI에서 이어진다.
3. 작업 전이(stage TT). 목표 작업 혼합으로 이어 가는데, 보통 제품에 따라 멀티 이미지나 비디오 쪽에 더 무겁다. 배포(deployment)를 위한 선택적 파인튜닝(fine-tune).

핵심은 커리큘럼 순서가 중요하다는 점이다. 비디오 먼저 또는 멀티 이미지 먼저 학습하면, 같은 데이터라도 단일 이미지 먼저보다 이미지 성능이 더 나쁘다. 논문은 이를 명시적으로 절제(ablate)한다.

### 커리큘럼이 작동하는 이유 (Why curriculum works)

단일 이미지 학습이 지각의 토대를 쌓는다. 패치 토큰이 세밀한 시각 특성(feature)을 운반하고, LLM이 그것을 텍스트와 통합하는 법을 배운다. 멀티 이미지와 비디오는 구조적 도전(어느 이미지가 어느 것인지, 무엇이 먼저 일어났는지)을 도입하는데, 이는 강한 지각 토대 없이는 배우기 어렵다.

모든 시나리오를 함께 밑바닥부터 학습하면, 모델은 지각을 과소적합(underfitting)하고(배치당 제한된 단일 이미지 데이터) 구조를 과적합(overfitting)한다(많은 멀티 이미지 / 비디오 데이터). 그 결과는 이미지 간 추론 패턴은 따르지만 시각적으로는 얕은 모델이다.

커리큘럼 순서는 stage SI에서 지각 강도를 주고, 그다음 stage OV에서 조합적·시간적 추론을 주되, 둘 중 어느 것도 잃지 않는다.

### 창발적 시나리오 간 기술 (Emergent cross-scenario skills)

LLaVA-OneVision 논문은 세 가지 창발 능력을 보고한다.

1. 멀티 카메라 추론. 멀티 이미지와 비디오로 따로 학습되었는데, 추론 시 멀티 카메라 주행 장면을 추론하라는 요청을 받는다. 모델은 학습에서 정확히 그 포맷을 본 적이 없는데도 시점들을 올바르게 통합한다.
2. 마크 집합(set-of-mark) 프롬프팅. 사용자가 이미지의 객체에 번호 매겨진 마크로 주석(annotation)을 단다. 모델은 "마크 3이 마크 7에 대해 무엇을 하고 있는지" 추론한다. 마크에도 주석에도 학습되지 않았으나, 공간적 그라운딩(grounding)과 멀티 이미지 참조의 조합에서 배웠다.
3. 아이폰 스크린샷 에이전트. 사용자가 아이폰 화면의 스크린샷을 제공하고 다음 클릭을 계획하라고 요청한다. UI 스크린샷, 사용자 워크플로의 비디오, 멀티 이미지 전후 쌍으로 학습되었다. 에이전트 사용 사례로 일반화한다.

이것들은 학습된 작업이 아니다. 커리큘럼의 조합적 구조에서 창발한다.

### 시각 토큰 풀링 (Visual-token pooling)

토큰 예산은 풀링을 요구한다. OneVision은 2D 패치 격자에 쌍선형 보간(bilinear interpolation)을 쓴다. 24x24 = 576개 패치가 12x12 = 144(2배 인수) 또는 8x8 = 64(3배 인수)가 된다. 풀링은 지역성(locality)을 보존하려고 토큰 공간이 아니라 패치 격자 공간에서 이루어진다.

시나리오별 풀링 인수를 무엇으로 고를지 자체가 하이퍼파라미터(hyperparameter)다. 풀링이 적을수록 토큰이 많아져 표현이 풍부해진다. 풀링이 많을수록 토큰이 적어져 더 많은 프레임이나 이미지가 들어간다.

### LLaVA-OneVision-1.5

2025년 후속작(LLaVA-OneVision-1.5, arXiv 2509.23661)은 학습 데이터, 모델 가중치, 코드에서 "완전 공개(fully open)"다. 일부 벤치마크에서 비공개와의 간극을 맞추고 레시피를 민주화한다. 같은 커리큘럼, 더 많은 데이터, 더 나은 베이스 LLM이며, 아키텍처 변경은 없다.

### Qwen2.5-VL과의 대비 (Contrast with Qwen2.5-VL)

Qwen2.5-VL(Lesson 12.09)은 다른 선택을 한다. 고정 풀링 대신 M-RoPE와 동적 FPS(dynamic FPS)를 쓴다. 그 예산은 입력에 따라 스케일링된다. 1분짜리 비디오가 5초짜리 비디오보다 더 많은 토큰을 쓴다. LLaVA-OneVision은 예산을 고정하고 풀링을 스케일링한다. 둘 다 동작하며, 설정 가능성과 예측 가능성을 맞바꾼다.

## 라이브러리로 써보기 (Use It)

`code/main.py`는 OneVision 스타일 VLM을 위한 커리큘럼이자 예산 계획기다. 샘플당 토큰 예산과 목표 시나리오 혼합(가령 단일 이미지 40%, 멀티 이미지 30%, 비디오 30%)이 주어지면, 이 코드는 다음을 한다.

- 시나리오별로 해상도, 풀링 인수, 프레임 개수를 할당한다.
- 모든 시나리오가 공유 예산 안에 맞는지 확인한다.
- 예상 토큰 개수, LLM FLOPs, 그리고 어느 시나리오가 토큰 부족(under-tokenized)인지 보고한다.
- 단계별 학습 스케줄을 출력한다.

OneVision 파인튜닝을 계획하거나 VLM 배포의 요청당 비용을 점검하는 데 사용하라.

## 산출물 (Ship It)

이 레슨은 `outputs/skill-onevision-budget-planner.md`를 만든다. 목표 작업 분포와 샘플당 예산이 주어지면, AnyRes 인수, 프레임별 풀링, 비디오 프레임 개수, 커리큘럼 단계 가중치를 내보낸다. 통합 시나리오 VLM을 학습하거나 파인튜닝할 때마다 이것을 사용하라.

## 연습 문제 (Exercises)

1. 당신의 제품은 단일 이미지 80%, 멀티 이미지 10%(2-4개 이미지), 비디오 10%(8-16개 프레임)를 지원한다. 토큰 예산을 설계하라. 무거운 멀티 이미지를 하지 않아서 절약한 여분 예산을 어디에 두겠는가?

2. LLaVA-OneVision Section 4.3(창발 능력)을 읽어라. 커리큘럼이 풀어낼 가능성이 높지만 논문이 보고하지 않은 네 번째 창발 기술을 제안하라.

3. 커리큘럼 순서를 바꿔라 — 멀티 이미지 먼저, 그다음 단일 이미지, 그다음 비디오를 학습하라. 어떤 벤치마크가 저하되는지, 왜인지 예측하라.

4. 논문은 샘플당 단 8프레임으로 학습된 비디오 벤치마크를 보고한다. 그것이 추론 시 30초짜리 비디오로 일반화되는가? 무엇이 먼저 망가지는가 — 토큰 예산인가 시간적 추론인가?

5. 24x24 패치를 12x12로 쌍선형 풀링하는 것은 차원당 4배 축소다. 표준 라이브러리 Python으로 풀링을 구현하고, 각 2x2 블록에 대한 평균이 쌍선형 출력과 일치하는지 검증하라.

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|-----------------|------------------------|
| OneVision 시나리오(OneVision scenario) | "단일 이미지, 멀티 이미지, 또는 비디오" | 통합 VLM이 처리하는 세 가지 입력 형상 중 하나. 예산은 가로질러 일정하게 유지된다 |
| 토큰 예산(Token budget) | "샘플당 토큰 몇 개" | 학습 / 추론 샘플당 LLM이 보는 총 시각 토큰, 보통 3000-4000 |
| 커리큘럼(Curriculum) | "학습 순서" | 창발적 전이를 위해 선택된 단계 순서(단일 이미지 → 멀티 이미지 → 비디오) |
| 쌍선형 풀링(Bilinear pooling) | "토큰 축소" | 지역성을 보존하면서 토큰 개수를 줄이기 위해 패치 격자(2D)에 쌍선형 보간을 적용 |
| 창발 기술(Emergent skill) | "학습 안 했는데도 됨" | 커리큘럼 조합 덕분에, 일치하는 학습 데이터 없이 추론 시 나타나는 능력 |
| AnyRes-k | "k-타일 설정" | 고정 해상도의 k개 서브 타일에 썸네일 하나를 더한 것, 전형적으로 k ∈ {4, 9} |
| 작업 전이(Task transfer) | "시나리오 간 일반화" | 공유 백본(backbone)을 통해 단일 이미지에서 배운 기술이 비디오에 적용됨(그리고 그 반대) |
