# 옴니 모델: Qwen2.5-Omni와 Thinker-Talker 분할

> 2024년 5월 GPT-4o의 제품 데모가 파괴적이었던 까닭은 기반 모델이 아니라 제품의 형태에 있었다. 말을 걸면 모델이 카메라가 보는 것을 보고 250ms 이내에 말로 대답하는 음성 인터페이스다. 오픈 생태계는 2024년과 2025년의 나머지 기간을 그 제품 표면에 도달하려는 경쟁에 쏟았다. Qwen2.5-Omni(2025년 3월)는 기준이 되는 오픈 설계다: Thinker(거대 텍스트 생성 트랜스포머)에 Talker(병렬 음성 생성 트랜스포머)를 더하고, 스트리밍 음성 토큰(token)으로 연결한다. Mini-Omni는 이를 단순화했고, Moshi는 그 지연 시간(latency)에 필적했으며, GLM-4-Voice는 이를 중국어로 확장했다. 이 레슨은 Thinker-Talker 아키텍처와, 스트리밍 실시간 대화를 동작하게 만드는 지연 시간 예산을 읽는다.

**Type:** Build
**Languages:** Python (stdlib, streaming pipeline latency simulator + VAD loop)
**Prerequisites:** Phase 12 · 19 (audio-LLMs), Phase 12 · 16 (any-to-any)
**Time:** ~180분

## 학습 목표 (Learning Objectives)

- 추론 파이프라인(pipeline)을 Thinker(텍스트 추론)와 Talker(음성 합성)로 분할하고 병렬 스트리밍이 동작하는 이유를 설명하기.
- 대화형 상호작용에 대한 첫 오디오 바이트까지의 시간(time-to-first-audio-byte, TTFAB) 예산을 구성 요소별로 계산하기.
- Thinker 내부에서 비전, 오디오, 텍스트에 걸친 TMRoPE의 시간 정렬 위치 인코딩(position encoding)을 설명하기.
- 세 가지 실시간 대화 패턴을 명명하기: 반이중(half-duplex), 턴 교대(turn-taking), 전이중(full-duplex).

## 문제 (The Problem)

실시간 음성 비서는 많은 일을 빠르게 해야 한다:

1. 사용자의 말을 듣는다. 실시간 음성 토큰화, 사용자가 말을 마쳤는지 알기 위한 음성 활동 탐지(voice activity detection, VAD).
2. 선택적으로 본다. 2~4 FPS의 카메라 입력을 오디오와 함께 Thinker로 스트리밍한다.
3. 생각한다. 대화 이력을 조건으로 응답을 구성한다.
4. 말한다. 오디오 토큰을 합성하고, 파형(waveform)으로 디코드하고, 사용자의 스피커로 스트리밍한다.

각 단계는 지연 시간을 더한다. 대화하는 느낌을 주려면 전체 왕복(round-trip)이 500ms 미만이어야 한다. 그 아래로 내려가면 사용자는 지연을 알아채지 못한다. GPT-4o는 약 250ms를 주장한다. Moshi는 약 160ms. Qwen2.5-Omni는 약 350~500ms다.

모든 구성 요소가 스트리밍되어야 한다. "전부 배치(batch)한 다음 디코드"는 어떤 것도 허용되지 않는다.

## 개념 (The Concept)

### Thinker와 Talker

Qwen2.5-Omni의 분해:

- Thinker: 7B~80B 텍스트 생성 트랜스포머. 인터리브된 텍스트 + 이미지 + 오디오 토큰을 소비한다. 무엇을 말할지를 나타내는 텍스트 토큰을 출력한다.
- Talker: 더 작은 음성 생성 트랜스포머(200M~1B). Thinker의 텍스트 출력 토큰과 최근 음성 컨텍스트 토큰을 소비한다. 이산 음성 토큰(잔차 VQ 인덱스)을 출력한다.
- 음성 디코더: 음성 토큰을 실시간으로 오디오 샘플로 바꾸는 스트리밍 파형 디코더(SNAC, MoVQGAN 계열).

이 분리가 중요하다. Thinker는 좋은 추론을 위해 커야 한다. Talker는 그 일이 국소적이기 때문에 — 텍스트를 음성 토큰으로 변환 — 작아도 된다. Talker가 크다고 더 표현력이 좋아지지는 않는다. 더 느려질 뿐이다.

둘을 병렬로 실행:

1. Thinker가 텍스트 토큰 t_i를 내보낸다.
2. Talker가 (스트리밍으로) t_i를 소비하고 음성 토큰 s_i, s_{i+1}, ..., s_{i+k}를 내보낸다.
3. 음성 디코더가 들어오는 대로 음성 토큰을 소비하고 오디오 샘플을 내보낸다.
4. Thinker가 텍스트 토큰 t_{i+3}에 도달할 무렵, Talker는 이미 t_0..t_{i+2}에 대한 오디오를 스트리밍했다.

### TMRoPE — 시간 정렬 멀티모달 위치

Thinker는 이미지 프레임(예컨대 4 FPS로 도착), 오디오 프레임(초당 50 프레임으로 도착), 대화 이력의 텍스트를 통합해야 한다. 순진한 시퀀스 순서(모든 이미지, 그다음 모든 오디오, 그다음 텍스트)는 시간 정렬을 잃는다.

TMRoPE는 모든 토큰에 절대 타임스탬프를 부여한다. t=2.3s의 비전 토큰. t=2.32s의 오디오 토큰. 사용자의 "멈춰"라는 t=2.35s의 텍스트 토큰. RoPE가 타임스탬프로 어텐션(attention)을 회전시키며, 모델은 이들을 시간적으로 동시인 것으로 본다.

이것이 "그가 인사를 하며 손을 흔들었다"를 동작하게 하는 인프라다. 모델은 영상 프레임과 오디오를 동일한 개념적 순간에 본다.

### 스트리밍 음성 합성

음성 토큰은 스트리밍되어야 한다. Mini-Omni(Xie & Wu, 2024)는 "언어 모델은 스트리밍으로 생각하면서 듣고 말할 수 있다"를 도입했다: Thinker 출력 토큰과 Talker 출력 토큰이 동일한 시퀀스 안에서 인터리브된다. Talker는 Thinker가 다음 텍스트 토큰을 확정하자마자 발화한다. 배치 경계가 없다.

Moshi(Défossez et al., 2024년 10월)는 가장 빠른 오픈 구현이다. 단일 A100에서 160ms TTFAB. 아키텍처는 교대하는 위치에서 텍스트 토큰과 음성 토큰을 내보내는 단일 7B 트랜스포머로, 생각 스트림과 말하기 스트림을 분리하는 "내부 독백(inner monologue)"을 갖춘다. 이는 사실상 Thinker + Talker를 신중한 학습으로 하나의 모델에 융합한 것이다.

### VAD와 턴 교대

음성 활동 탐지는 입력 측에서 실행된다. 두 가지 패턴:

- 반이중(Half-duplex): 사용자가 말하면 모델이 듣는다. 모델이 말하면 사용자가 듣는다. VAD 침묵 탐지(약 200ms)로 명확하게 인계한다.
- 전이중(Full-duplex): 둘이 동시에 말할 수 있다. 모델이 맞장구("어-허")를 치거나 끼어들 수 있다. 훨씬 어렵다. Moshi가 이를 지원한다.

Qwen2.5-Omni는 기본적으로 반이중을 지원하며, 침묵 임계값으로 턴 교대를 한다. 전이중은 애플리케이션 계층의 처리가 필요하다.

### Qwen3-Omni (2025년 11월)

후속작. Qwen3-80B Thinker, 더 큰 Talker, 개선된 TMRoPE-v2. 지연 시간이 GPT-4o의 250ms에 근접한다. 오픈 가중치(weight). OmniBench 벤치마크에서 Gemini 2.0 Live와 경쟁적이다.

### 프로덕션 지연 시간 예산

전형적인 스트리밍 상호작용의 경우:

- 마이크 -> 오디오 토큰: 40~80ms.
- 프리필(prefill)(프롬프트 + 이력): 7B에서 100~200ms, 70B에서는 훨씬 더.
- 첫 Thinker 텍스트 토큰: 40ms.
- Talker가 첫 텍스트 토큰을 처리: 20ms.
- 첫 음성 토큰 확정: 40ms.
- 잔차 VQ 디코드: 30ms.
- 음성 파형 디코드: 50~80ms.

총 TTFAB: 7B에서 320~510ms, 70B에서 600~900ms. 프런티어 품질은 보통 70B 이상을 의미하며, 그래서 프런티어 지연 시간 격차가 발생한다.

### 토큰 레이트 계산

50 Hz 기본 음성 토큰을 쓰는 16kHz 음성에서는 출력 1초당 50개의 음성 토큰이 필요하다. Talker는 따라가려면 초당 50토큰 이상을 내보내야 한다. H100에서 전형적인 LLM 처리량(throughput)이 초당 30~80토큰임을 고려하면, 작은(200~300M) Talker는 충분히 빠르지만, 7B Talker는 뒤처진다.

이것이 "그냥 메인 모델을 쓰면 되지" 대신 작은 전용 Talker 모델이 존재하는 이유다.

## 라이브러리로 써보기 (Use It)

`code/main.py`:

- 모의 토큰 방출 속도로 Thinker-Talker 파이프라인을 시뮬레이션한다.
- 설정 가능한 모델 크기와 마이크 샘플 레이트에 대한 TTFAB를 계산한다.
- VAD 침묵 임계값으로 반이중 턴 교대를 시연한다.

## 산출물 (Ship It)

이 레슨은 `outputs/skill-omni-streaming-budget.md`를 생성한다. 실시간 음성 제품의 목표 TTFAB와 기능 집합(비전 입력, 이중 언어, 전이중)이 주어지면, Qwen2.5-Omni, Qwen3-Omni, Moshi, 또는 Mini-Omni를 선택하고 Thinker/Talker의 크기를 정한다.

## 연습 문제 (Exercises)

1. 목표 TTFAB가 300ms다. 7B Thinker와 300M Talker에서 모든 구성 요소의 지연 시간을 적어 보라.

2. Qwen2.5-Omni는 TMRoPE를 사용한다. 사용자가 t=1s에 말을 시작하고 카메라가 t=1.2s에 제스처를 포착하는 프롬프트에 대해 모델이 무엇을 보는지 설명하라.

3. 전이중 지원은 모델이 들으면서 오디오를 내보내야 한다. 이를 가르치는 학습 데이터 형식을 제안하라.

4. Moshi 논문의 4절을 읽으라. "내부 독백" 분리와 그것이 Thinker-Talker 분할을 피하는 이유를 설명하라.

5. 처리량 예산을 계산하라: 50개의 기본 계층 토큰/초의 16kHz 음성을 따라가려면 Talker가 얼마나 빠르게 토큰을 내보내야 하는가?

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|-----------------|------------------------|
| Thinker | "추론하는 뇌" | 무엇을 말할지를 생성하는 거대 텍스트 생성 트랜스포머 |
| Talker | "음성을 생성하는 입" | Thinker의 텍스트로부터 이산 음성 토큰을 생성하는 작은 트랜스포머 |
| TTFAB | "지연 시간 예산" | 첫 오디오 바이트까지의 시간: 사용자 발화 종료부터 첫 오디오 샘플 출력까지 |
| TMRoPE | "시간 정렬 RoPE" | 비전, 오디오, 텍스트에 걸쳐 절대 타임스탬프를 사용하는 위치 인코딩 |
| 반이중(Half-duplex) | "턴 교대" | 사용자와 모델이 번갈아 함; VAD 침묵이 사용자 종료를 탐지 |
| 전이중(Full-duplex) | "동시" | 모델이 동시에 말하고 들을 수 있음; 맞장구 가능 |
| 내부 독백(Inner monologue) | "Moshi 분리" | 생각 스트림과 말하기 스트림이 인터리브되는 단일 모델 설계 |
