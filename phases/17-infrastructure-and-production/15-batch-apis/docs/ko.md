# 배치 API(Batch APIs) — 업계 표준이 된 50% 할인

> 모든 주요 프로바이더가 50% 할인과 ~24시간 처리 시간을 갖춘 비동기 배치 API를 출시한다. OpenAI, Anthropic, Google, 그리고 대부분의 추론(inference) 플랫폼(Fireworks 배치 티어, Together 배치)이 같은 패턴을 구현한다. 배치를 프롬프트 캐싱과 쌓으면 야간 파이프라인은 동기·비캐시 비용의 ~10%로 떨어진다. 규칙은 잔인하리만치 단순하다: 대화형(interactive)이 아니라면 배치에 속한다. 콘텐츠 생성 파이프라인, 문서 분류(classification), 데이터 추출, 리포트 생성, 대량 레이블링, 카탈로그 태깅 — 24시간 지연 시간(latency)을 견딜 수 있는 무엇이든 배치로 옮기기 전까지는 테이블에 두고 온 돈이다. 2026년 프로덕션 패턴은 모든 새 LLM 워크로드를 세 레인으로 분류하는 것이다: 대화형(캐싱이 있는 동기), 준대화형(폴백이 있는 비동기 큐), 배치(야간, 캐시된 입력 쌓기). 대화형인 척하지만 수 분의 지연 시간을 견디는 워크로드가 가장 많이 낭비한다.

**Type:** Learn
**Languages:** Python (stdlib, toy batch-vs-sync cost simulator)
**Prerequisites:** Phase 17 · 14 (Prompt & Semantic Caching)
**Time:** ~45분

## 학습 목표 (Learning Objectives)

- 세 프로바이더 배치 API(OpenAI, Anthropic, Google)와 공통의 50% 할인 + 24시간 처리 시간 보장을 짚기.
- 야간 분류 워크로드에서 배치 + 캐시된 입력 쌓기의 비용을 계산하고 동기·비캐시 베이스라인과 비교하기.
- 워크로드를 대화형 / 준대화형 / 배치로 분류하고 그 레인을 정당화하기.
- 두 함정을 짚기: 부분 대화성(사용자가 24시간보다 빠른 것을 기대)과 출력 스키마 드리프트(배치 파일 형식이 프로바이더마다 다름).

## 문제 (The Problem)

당신의 팀은 야간 리포트 생성 파이프라인을 출시한다. 문서 50,000개, 각각 요약하고, 요약을 클러스터링하고, 경영진 브리프를 초안한다. 동기로 실행하면 밤마다 $2,000에 4시간이 걸린다. 당신은 배치 API에 대해 듣는다.

배치는 50% 할인을 준다. 또한 (50k 호출 전체에 공유되는) 시스템 프롬프트에 프롬프트 캐싱을 켠다. 쌓으면 청구액은 밤마다 $180로 떨어진다 — 베이스라인의 ~9%. 같은 파이프라인, 세 가지 설정 변경.

배치는 아무도 당기지 않는, LLM 비용 도구함에서 가장 저렴한 레버다. 그 이유는 대개 조직적이다: SLA가 실제로는 "아침까지"인데 팀은 "실시간"이라고 생각한다. 이 레슨은 청구액의 90%를 테이블에 두고 오지 않는 것에 관한 것이다.

## 개념 (The Concept)

### 세 배치 API

**OpenAI Batch API**: 요청 목록과 함께 JSONL 파일 업로드. 약속된 24시간 처리 시간(실무에서는 보통 ~2-8시간). 입력 및 출력 토큰에 50% 할인. `/v1/batches` 엔드포인트. 캐시 적격 입력은 그 위에 캐시된 입력 가격도 받는다.

**Anthropic Message Batches**: JSONL 업로드. 24시간 처리 시간. 50% 할인. `cache_control` 지원 — 캐시 쓰기는 명시적이고, 읽기는 배치 내에서 자동으로 일어난다.

**Google Vertex AI Batch Prediction**: BigQuery 또는 GCS 입력. Gemini에 대해 유사한 50% 할인. Vertex 파이프라인과 통합.

### 의미론: 비동기이지, 느린 것이 아니다

배치는 "24시간 이내에 반환하기로 약속한다"이지 "이것은 24시간이 걸릴 것이다"가 아니다. 전형적 P50은 2-6시간이다. 프로바이더는 GPU 재고가 미활용되는 비피크 윈도우에 당신의 배치를 스케줄링한다.

### 캐싱과 쌓기

같은 4K 토큰 시스템 프롬프트를 쓰는 50k 문서 요약:

- 동기 비캐시: 50000 × ($input × 4000 + $output × 200), 전체 요율로.
- 동기 캐시: 첫 쓰기 후 시스템 프롬프트 캐시됨. 나머지 49999는 10배 저렴한 입력을 받음.
- 배치 캐시: 위의 모든 것에 더해 읽기와 쓰기 양쪽에 50% 할인.

쌓기: 배치 + 캐시 = 동기 비캐시 청구액의 ~10%. 야간에 돌고 공유 시스템 프롬프트를 가진 어떤 워크로드든 이것을 써야 한다.

### 워크로드 분류

**대화형(Interactive)** — 사용자가 응답을 기다린다. TTFT가 중요하다. 프롬프트 캐싱이 있는 동기 호출. 배치 불가.

**준대화형(Semi-interactive)** — 사용자가 작업을 제출하고 몇 분 후 다시 확인한다. 배치가 없으면 동기로 폴백하는 비동기 큐. 중간 볼륨의 RAG 인덱싱을 생각하라.

**배치(Batch)** — 사용자가 "아침까지" 또는 "다음 시간까지" 결과를 기대한다. 콘텐츠 파이프라인, 대규모 분류, 오프라인 분석. 항상 배치하고, 항상 캐싱을 쌓는다.

흔한 실수: 파이프라인이 프로덕션이라는 이유로 모든 것을 대화형으로 분류하는 것. 프로덕션은 지연 시간 사양이 아니다 — SLA가 그렇다.

### 부분 대화성 함정

일부 기능은 대화형으로 보이지만 5-10분을 견딘다. 예: "새로 고침" 버튼이 있는 야간 고객 건강 리포트. 사용자가 새로 고침을 클릭하고 10분 기다리는 것은 괜찮다. 팀은 이를 동기로 출시한다. 50개의 동시 새로 고침은 배치되어 이메일로 전달되는 것보다 10배 비용이 든다.

물어야 할 질문: "이 사용자에게 24시간이 무엇을 의미하는가?" 답이 "그들은 알아차리지 못할 것이다"라면, 배치하라.

### 출력 스키마 함정

배치 파일 형식은 프로바이더마다 다르다:

- OpenAI: JSONL, 한 줄당 한 요청.
- Anthropic: JSONL, 한 줄당 한 메시지; 응답 형식 내장.
- Vertex: BigQuery 테이블 또는 TFRecord가 있는 GCS 프리픽스.

프로바이더 전반에 "하나의 배치 클라이언트"를 작성한다는 것은 프로바이더별 어댑터 코드를 뜻한다. 멀티 프로바이더 배치를 광고하는 게이트웨이(Portkey, 일부 티어의 LiteLLM)도 여전히 원시 형식을 얇게 래핑한다.

### 기억해야 할 숫자들

- 프로바이더 전반의 배치 할인: 입력 + 출력에 일률 50%.
- 처리 시간 SLA: 24시간 보장, 2-6시간 전형적 P50.
- 쌓인 배치 + 캐시된 입력: 동기 비캐시 비용의 ~10%.
- 워크로드 분류 규칙: 24시간 지연 시간이 허용되면, 항상 배치.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 3K 토큰 시스템 프롬프트와 500 토큰 출력을 가진 100k 문서 파이프라인에 대해, 전체 스택(배치 + 캐시) 대 동기 베이스라인의 절감액을 계산하라.
2. 당신이 아는 실제 제품에서 세 기능을 고르라. 각각을 대화형/준/배치로 분류하라.
3. 한 사용자가 리포트에 3시간이 걸렸다고 불평한다. 그것은 배치 오분류였는가 아니면 정당한 대화형이었는가? 결정 기준을 작성하라.
4. 당신의 배치 API 반환 SLA는 24시간이지만 P99는 20시간이다. 이것을 사용자에게 어떻게 전달할 것인가 — 엣지 케이스에서 하류 시스템 동작은 무엇인가?
5. 손익분기점을 계산하라: 어느 공유 프리픽스 길이에서 배치 + 캐시가 자체 예약 GPU에서 야간에 돌리는 것보다 저렴해지는가?

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|------|----------------|------------------------|
| 배치 API (Batch API) | "비동기 할인" | 24시간 처리 시간으로 50% 할인 |
| JSONL | "배치 형식" | 한 줄당 한 JSON 요청; OpenAI/Anthropic 표준 |
| Message Batches | "Anthropic 배치" | Anthropic의 배치 API 제품명 |
| Batch prediction | "Vertex 배치" | Vertex AI의 배치 API 제품 |
| 처리 시간 SLA (Turnaround SLA) | "24시간 약속" | 전형값이 아니라 보장; 전형값은 2-6시간 |
| 워크로드 분류 (Workload triage) | "대화성 결정" | 대화형 / 준 / 배치 라우팅 결정 |
| 출력 스키마 (Output schema) | "응답 형식" | 프로바이더별 JSONL 레이아웃; 이식 불가 |
| 쌓인 할인 (Stacked discount) | "배치 + 캐시" | 둘 다 적용 시 비캐시 동기 청구액의 ~10% |
