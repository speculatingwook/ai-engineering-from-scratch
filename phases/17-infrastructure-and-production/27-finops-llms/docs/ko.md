# LLM을 위한 FinOps — 단위 경제와 멀티테넌트 귀속

> 전통적 FinOps는 LLM 지출에서 무너진다. 비용은 리소스 가동 시간이 아니라 토큰(token) 트랜잭션이다. 태그(tag)가 매핑되지 않는다 — API 호출은 자산이 아니라 트랜잭션이다. 엔지니어링 결정(프롬프트 설계, 컨텍스트 윈도(context window), 출력 길이)이 곧 재무 결정이다. 2026년 플레이북은 첫날에 계측해야 할 세 가지 귀속(attribution) 차원을 가진다: 좌석 가격(seat pricing)과 확장을 위한 사용자별(`user_id`), 제품 표면 비용과 우선순위를 위한 과제별(`task_id` + `route`), 단위 경제(unit economics)와 갱신을 위한 테넌트별(`tenant_id`). 네 가지 토큰 계층 — 프롬프트, 도구(tool), 메모리, 응답 — 하나의 버킷은 지출을 숨긴다. 멀티테넌트(multi-tenant) 제품을 위한 집행 사다리(enforcement ladder): 테넌트당 속도 제한(rate limit)(예상 피크의 2~3배, 명확한 429 + retry-after); 일일 지출 상한(spend cap)(계약 상한의 1.5~3배; 속도 강화 + 경보 발동); 지출 z-점수 > 4에서 킬 스위치(kill switch)(자동 일시정지 + 온콜 호출). 귀속 패턴: 태그-후-집계, 텔레메트리 조이너(telemetry-joiner)(트레이스 ID → 청구; 최고 정확도), 샘플링-후-외삽, 모델 기반 할당, 이벤트 소싱, 실시간 스트리밍. 단위 지표: 해결된 쿼리당 비용, 생성된 산출물당 비용 — $/M 토큰이 아니라. 소급(retroactive) 태깅은 항상 놓친다; 요청 생성 시점에 계측하라.

**Type:** Learn
**Languages:** Python (stdlib, toy cost-attribution simulator with kill switch)
**Prerequisites:** Phase 17 · 13 (Observability), Phase 17 · 14 (Caching)
**Time:** ~60분

## 학습 목표 (Learning Objectives)

- 전통적 FinOps(태그 + 등급)가 LLM 지출에서 무너지는 이유를 설명하고 세 가지 새 귀속 차원을 명명하기.
- 네 가지 토큰 계층(프롬프트, 도구, 메모리, 응답)을 열거하고 단일 버킷 청구가 비용을 숨기는 이유를 설명하기.
- 멀티테넌트 제품을 위한 집행 사다리(속도 → 지출 상한 → 킬 스위치)를 설계하기.
- $/M 토큰 대신 단위 지표(해결된 쿼리 / 산출물당 비용)를 고르기.

## 문제 (The Problem)

청구서에 $40,000이 찍혀 있다. 당신은 모른다:
- 어느 테넌트가 그것을 썼는지.
- 어느 제품 기능이 그것을 일으켰는지.
- 어느 개별 사용자가 남용했는지.
- 프롬프트 부풀림(bloat), 도구 호출, 메모리 증폭 중 무엇이 범인인지.

프로바이더 측의 태그-후-집계는 태그가 라인 항목(line item)으로 전파되는 클라우드 리소스(EC2, S3)에는 통한다. LLM API 호출은 자동 태그되지 않는다 — 호출 지점(call site)에서 사용자/과제/테넌트를 찍고 끝까지 운반해야 한다. 소급 귀속은 항상 엣지 케이스를 놓친다.

## 개념 (The Concept)

### 세 가지 귀속 차원

**사용자별**(`user_id`): 누가 무엇을 쓰고 있는가. 좌석 가격, 확장 대화를 이끌고 파워 유저를 식별한다.

**과제별**(`task_id` + `route`): 어느 제품 표면이 무엇을 쓰는가. 기능 우선순위, 비싼 기능 제거 결정을 이끈다.

**테넌트별**(`tenant_id`): 어느 고객이 수익성이 있는가. 단위 경제, 갱신 가격, 등급 임계값을 이끈다.

세 가지 모두 첫날에 호출 지점에서 계측하라. 소급은 항상 더 나쁘다.

### 네 가지 토큰 계층

| 계층 | 예시 | 전형적 총합 비율 |
|-------|---------|---------------------|
| 프롬프트 | 시스템 + 사용자 입력 | 40~60% |
| 도구 | 되먹임된 도구 호출 결과 | 20~40%(에이전트 워크로드) |
| 메모리 | 이전 대화 / 검색된 문서 | 10~30% |
| 응답 | 모델 출력 | 10~30% |

네 가지를 함께 버킷에 넣으면 최적화가 눈멀게 된다. 귀속 스키마에서 분리하라.

### 집행 사다리

1. **속도 제한** 테넌트당. 예상 피크의 2~3배. `Retry-After`와 함께 429를 반환. 테넌트는 마찰을 보지만 놀랄 청구서는 없다.

2. **일일 지출 상한** 테넌트당. 계약 상한의 1.5~3배. 발동: 속도 제한 강화 + 고객 성공(customer-success) 부서 경보.

3. **킬 스위치** 테넌트 베이스라인 대비 지출 z-점수 > 4에서. 테넌트 자동 일시정지; 온콜 호출; 운영 + CS로 에스컬레이션.

### 귀속 패턴

- **태그-후-집계**: 메타데이터 헤더를 찍고; 나중에 집계. 단순; 거침.
- **텔레메트리 조이너**: 트레이스 ID를 통해 트레이스를 청구에 조인. 최고 정확도. 성숙한 팀이 하는 것.
- **샘플링 + 외삽**: 5~10% 샘플링 후 곱하기. 대략적 지출에 비용 효율적; 꼬리(tail)를 놓침.
- **모델 기반 할당**: 비용 동인을 추론하기 위한 회귀. 태그 없는 레거시 데이터용.
- **이벤트 소싱**: 비용을 스트림(Kafka / Kinesis)의 이벤트로. 실시간.
- **실시간 스트리밍**: 1초 미만으로 대시보드 갱신.

### X당 비용이 단위 지표다

$/M 토큰은 벤더 화법이다. 제품 지표:

- 해결된 지원 티켓당 비용.
- 생성된 기사당 비용.
- 성공한 에이전트 과제당 비용.
- 사용자-세션-분당 비용.

비용을 제품 성과에 묶어라. 그렇지 않으면 최적화가 고정점 없이 떠돈다.

### 비용 귀속 트레이스 형태

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

모든 호출에서 방출(emit)하라. 데이터 레이크(data lake)에 저장하라. 차원별로 집계하라. Phase 17 · 13 관측성(observability) 스택이 이것이 사는 곳이다.

### 복리 절감 스택

스택: 캐시 + 배치 + 라우트 + 게이트웨이. 네 가지 모두 적용 시:
- 캐시 L2(Phase 17 · 14): 입력 약 10배 저렴.
- 배치(Phase 17 · 15): 50% 할인.
- 저렴한 모델로 라우트(Phase 17 · 16): 60% 비용 절감.
- 게이트웨이 효율(Phase 17 · 19): 중복성 + 재시도.

최선의 경우 스택: 순진한 베이스라인의 약 5~10%. 대부분의 팀은 2~3개 레버를 작동시킨다; 네 가지 모두 쌓는 팀은 드물다.

### 기억해 둘 숫자들

- 귀속 차원: 사용자별, 과제별, 테넌트별.
- 네 가지 토큰 계층: 프롬프트, 도구, 메모리, 응답.
- 킬 스위치: 지출 z-점수 > 4.
- 단위 지표: $/M 토큰이 아니라 해결된 쿼리당 비용.
- 쌓인 최적화: 베이스라인의 약 5~10% 가능.

2. 테넌트별, 과제별 비용 대시보드를 설계하라. 가장 먼저 만드는 5개 뷰는 무엇인가?

4. 지원 제품의 해결된 티켓당 비용을 계산하라: 티켓당 3M 토큰, 하루 약 800 티켓, GPT-5 캐시 요율.

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|------|----------------|------------------------|
| 사용자별 귀속 (Per-user attribution) | "사용자 수준 비용" | 모든 호출에 찍힌 `user_id` |
| 과제별 귀속 (Per-task attribution) | "기능 비용" | `task_id` + `route`가 제품 표면을 식별 |
| 테넌트별 귀속 (Per-tenant attribution) | "고객 비용" | `tenant_id`; 단위 경제를 이끔 |
| 네 가지 토큰 계층 (Four token layers) | "비용 계층" | 프롬프트 + 도구 + 메모리 + 응답 |
| 속도 제한 (Rate limit) | "429 가드" | 게이트웨이에서 집행되는 테넌트당 상한 |
| 일일 지출 상한 (Daily spend cap) | "일일 상한" | 경보를 동반한 테넌트 범위 예산 |
| 킬 스위치 (Kill switch) | "자동 일시정지" | 지출 z-점수 > 4가 자동 정지를 발동 |
| 해결당 비용 (Cost per resolved) | "제품 단위 지표" | 토큰이 아니라 제품 성과에 묶인 비용 |
| 텔레메트리 조이너 (Telemetry joiner) | "트레이스-청구 연결" | 최고 정확도 귀속 패턴 |
| 쌓인 최적화 (Stacked optimization) | "캐시+배치+라우트+게이트웨이" | 베이스라인 약 5~10%까지의 복리 절감 |
