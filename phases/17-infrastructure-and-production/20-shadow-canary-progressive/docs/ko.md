# LLM을 위한 섀도 트래픽, 카나리 롤아웃, 점진적 배포

> LLM 롤아웃은 소프트웨어 배포(deployment)에서 가장 어려운 부분들을 한데 모은 것이다. 단위 테스트가 없고 실패 양상(failure mode)이 분산되어 있으며 신호가 지연된다. 순서는 다음과 같다. (1) 섀도 모드(shadow mode) — 프로덕션(production) 요청을 후보 모델로 복제해 로그를 남기고, 사용자에게 아무 영향 없이 비교한다. 명백한 분포 문제는 잡아내지만 품질을 보장하지는 않는다. (2) 카나리 롤아웃(canary rollout) — 트래픽을 10% → 25% → 50% → 75% → 100%로 점진적으로 이동시키며 각 단계마다 게이트(gate)를 둔다. 지연 시간(latency) 백분위수, 요청당 비용, 오류/거부율, 출력 길이 분포, 사용자 피드백 비율을 추적한다. (3) 안정성이 확인된 뒤 서로 뚜렷이 다른 대안들에 대한 A/B 테스트. 비결정성(non-determinism)은 줄일 수 없다 — GPU 부동소수점 비결합성(FP non-associativity)에 배치 크기(batch-size) 분산이 더해져, 동일한 입력으로도 실행마다 최대 15%의 정확도 변동이 발생한다. 비용은 상수가 아니라 변수다 — 20% 더 나은 모델이 호출당 3배 더 비쌀 수 있다. 롤백(rollback) 속도가 결정적이다. 롤백에 재배포(redeploy)가 필요하다면 너무 느린 것이다. 정책(policy)은 설정/플래그에 두고, 모델은 고정된 다이제스트(digest)와 함께 레지스트리(registry)에 두면, 롤백 = 정책 뒤집기 + 임계값 되돌리기 + 옛 모델 고정을 몇 초 만에 끝낼 수 있다.

**Type:** Learn
**Languages:** Python (stdlib, toy canary-progression simulator)
**Prerequisites:** Phase 17 · 13 (Observability), Phase 17 · 21 (A/B Testing)
**Time:** ~60분

## 학습 목표 (Learning Objectives)

- 섀도 모드(영향 없는 비교), 카나리(실 트래픽 점진 노출), A/B(안정성 확인 후 비교)를 구별하기.
- LLM 특유의 다섯 가지 카나리 지표(지연 시간, 요청당 비용, 오류/거부, 출력 길이 분포, 사용자 피드백)를 열거하기.
- LLM 비결정성(최대 15%)이 롤아웃에서 "안정적"이라는 말의 의미를 어떻게 바꾸는지 설명하기.
- 시간(재배포)이 아니라 초(정책 뒤집기) 단위로 끝나는 롤백 경로 설계하기.

## 문제 (The Problem)

새 모델을 배포한다. 오프라인 평가(eval)는 정확도 3% 향상을 보여준다. 프로덕션에서 켠다. 24시간 안에 비용이 40% 오르고, 사용자의 부정 평가(thumbs-down)가 8% 늘고, 고객 티켓 세 건이 "이상한 답변"을 신고한다. 롤백한다. 재배포에 3시간이 걸린다. 주말이 망가졌다.

그 모든 것은 피할 수 있었다. 섀도 모드였다면 어떤 사용자가 보기 전에 40% 비용 급증을 잡아냈을 것이다. 카나리였다면 부정 평가가 움직였을 때 10%에서 멈췄을 것이다. 정책 플래그(policy-flag) 롤백이었다면 30초가 걸렸을 것이다. "오프라인 평가가 좋아 보인다"와 "실제 사용자가 만족한다" 사이의 간극을 메우는 것이 바로 이 규율(discipline)이다.

## 개념 (The Concept)

### 섀도 모드 (Shadow mode)

후보 모델이 프로덕션과 동일한 요청을 받는다. 출력은 로그에 남기되 사용자에게 반환하지 않는다. 사용자 영향은 0이다. 다음을 로그에 남긴다.

- 출력 내용(프로덕션과의 diff).
- 토큰(token) 수(비용 차이).
- 지연 시간.
- 거부 및 오류.

잡아내는 것: 비용 폭증, 길이 회귀(length regression), 명백한 거부 변화, 하드 에러(hard error). 잡아내지 못하는 것: 사용자가 체감할 품질 차이. 섀도는 스모크 테스트(smoke test)이지 품질 테스트가 아니다.

### 카나리 롤아웃 (Canary rollout)

게이트를 둔 점진적 트래픽 이동. 전형적인 진행: 1% → 10% → 25% → 50% → 75% → 100%. 각 단계에서 5개 지표에 대해 게이트를 건다.

1. **지연 시간 백분위수** — P50, P95, P99. 위반: 카나리의 P99 > 베이스라인(baseline)의 1.5배.
2. **요청당 비용** — 혼합 $. 위반: 베이스라인 대비 20% 초과.
3. **오류 / 거부율** — 5xx에 명시적 거부 포함. 위반: 베이스라인의 2배.
4. **출력 길이 분포** — 평균 + P99. 위반: 분포 이동(distributional shift).
5. **사용자 피드백 비율** — 부정 평가 / 티켓 제출. 위반: 베이스라인의 1.5배.

### 비결정성이 새로운 분산이다

동일한 입력이라도 출력이 똑같지 않다. 이유는 다음과 같다.

- GPU 부동소수점 비결합성(배치마다 부동소수점 리덕션 순서가 달라짐).
- 배치 크기 분산(128 배치 안의 동일 프롬프트 vs 16 배치 안의 동일 프롬프트).
- 샘플링(sampling)(온도 > 0).

측정 결과: 동일한 평가 셋에서 실행 간 최대 15%의 정확도 변동. 롤아웃에서 "안정적"이라는 말은 지표가 예상 분산 범위 안에 있다는 뜻이지, 베이스라인과 동일하다는 뜻이 아니다. 게이트를 잡음 바닥(noise floor) 위로 설정하라.

### 비용은 변수다

20% 더 나은 모델이 호출당 3배 더 비쌀 수 있다. 요청당 비용은 다섯 게이트 중 하나다. 단위 경제(unit economics)를 깨뜨리는 "더 나은" 모델을 배포하는 것은 롤백 사유다.

### 롤백이 무기다

- 정책 플래그(기능 플래그 시스템): 설정에서 비율을 뒤집는다. 몇 초가 걸린다.
- 모델 고정(레지스트리 다이제스트): 고정된 모델은 자동 업그레이드되지 않는다.
- 롤백 = 플래그 되돌리기 + 고정 다이제스트를 이전 것으로 설정. 시간이 아니라 초 단위.

스택이 롤백을 위해 재배포를 요구한다면, 롤아웃하기 전에 그것부터 고쳐라.

### 도구 (Tooling)

**Argo Rollouts** / **Flagger** — Kubernetes 점진적 전달(progressive delivery) 컨트롤러. Istio/Linkerd 가중 라우팅(weighted routing)과 통합된다.

**Istio 가중 라우팅** — 서비스 메시(service-mesh) 수준의 트래픽 분할.

**KServe / Seldon Core** — 카나리가 내장된 모델 서빙(serving).

**기능 플래그(Feature flags)** — LaunchDarkly, Flagsmith, Unleash. 정책 수준의 뒤집기, 재배포 없음.

### 지표 측정 주기 (Metrics cadence)

카나리 게이트는 트래픽 양에 따라 5~15분마다 점검한다. 분당 10 요청의 1% 트래픽이면 윈도(window)당 데이터 포인트가 50~150개 모인다 — 지연 시간에는 충분하지만 사용자 피드백에는 잡음이 많다. 10%면 약 10배로 늘어난다. 진행은 각 단계에서 충분한 샘플을 모을 만큼 길게 멈춰야 한다.

### A/B 단계는 선택이다

새 모델이 뚜렷하게 다르다면(다른 동작, 다른 비용 곡선, 다른 어조), 카나리를 통과한 뒤 50%에서 A/B 테스트하라. 단지 개선된 버전이라면, 카나리 게이트를 통과할 때 100%로 건너뛰어라.

### 기억해 둘 숫자들

- 카나리 진행: 1% → 10% → 25% → 50% → 75% → 100%.
- 비결정성 상한: 동일 입력에서 실행 간 최대 15% 분산.
- 다섯 카나리 지표: 지연 시간, 비용, 오류/거부, 출력 길이, 사용자 피드백.
- 비용 게이트: 베이스라인 대비 20% 초과는 위반.
- 롤백: 시간이 아니라 초 단위.

이 레슨은 `outputs/skill-rollout-runbook.md`를 만든다. 후보 모델, 베이스라인, 위험 허용도(risk tolerance)가 주어지면 섀도→카나리→100% 계획을 설계한다.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 25% 비용 회귀를 주입하라. 카나리는 어느 단계에서 멈추는가?
2. 새 모델이 오프라인에서 정확도 3% 향상을 보이지만 요청당 비용은 +18%다. 배포할 것인가? 정책에 따라 다르다 — 두 경로를 모두 작성하라.
3. 끝에서 끝까지 60초 미만이 걸리는 롤백을 설계하라. 필요한 인프라를 나열하라.
4. 비결정성이 평가에서 ±7%를 보인다. 거짓 경보가 나지 않도록 카나리 게이트를 설정하라. 어떤 배수를 쓰는가?
5. 섀도 모드가 카나리 전에 40% 비용 급증을 잡아낸다. 섀도에서 발동하는 경보 규칙을 작성하라.

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|------|----------------|------------------------|
| 섀도 모드 (Shadow mode) | "새 모델로 복제" | 로깅을 위해 후보 모델로 보내는, 영향 없는 전송 |
| 카나리 (Canary) | "점진적 트래픽" | 게이트를 둔, 점진적으로 사용자에게 노출되는 롤아웃 |
| 게이트 (Gates) | "롤아웃 점검" | 진행을 막는 지표 임계값 |
| 비결정성 (Non-determinism) | "LLM 분산" | 줄일 수 없는 실행 간 차이 |
| 정책 플래그 (Policy flag) | "플래그 뒤집기 롤백" | 설정 수준의 롤백, 시간이 아니라 초 단위 |
| 모델 고정 (Model pin) | "레지스트리 다이제스트" | 모델 버전에 대한 불변(immutable) 참조 |
| Argo Rollouts | "K8s 점진적" | Kubernetes 네이티브 카나리/롤백 컨트롤러 |
| KServe | "추론 K8s" | 카나리 기본 요소(primitive)를 갖춘 모델 서빙 |
| Istio 가중 라우팅 (Istio weighted) | "메시 분할" | 서비스 메시 트래픽 분할기 |
