# 챗봇에서 장기 지평 에이전트로의 전환

> 2023년의 챗봇은 한 번의 턴(turn)으로 질문에 답했다. 2026년의 프런티어 모델(frontier model)은 하나의 작업에 일상적으로 수 분에서 수 시간을 쓴다. METR의 Time Horizon 1.1 벤치마크(benchmark)(2026년 1월)는 Claude Opus 4.6을 50% 신뢰도에서 14시간 이상의 전문가 작업으로 평가한다. 이 지평(horizon)은 GPT-2 이후 대략 7개월마다 두 배로 늘어나 왔다. 단일 턴 채팅을 중심으로 우리가 세운 모든 가정 — 컨텍스트, 신뢰, 실패 양상, 비용, 관측 가능성 — 은 실행 시간이 점심시간보다 길어지는 순간 무너진다.

**Type:** Learn
**Languages:** Python (stdlib, horizon-curve simulator)
**Prerequisites:** Phase 14 · 01 (The Agent Loop)
**Time:** ~45분

## 문제 (The Problem)

챗봇(chatbot)은 상태 없는(stateless) 함수다. 프롬프트(prompt)를 받아 답변을 반환하고, 잊어버린다. 2024년까지 만들어진, RAG를 갖춘 시스템조차 이렇게 동작한다. 단일 컨텍스트 윈도우(context window) 안에서 계획하고, 한 번 행동하고, 결과를 내놓는다.

자율 에이전트(autonomous agent)는 종류 자체가 다르다. 루프(loop)를 돈다. 언제 멈출지 스스로 결정한다. 실행 도중에 돈을 쓴다 — 실제 토큰(token), 실제 GPU 시간, 실제 하류(downstream) 부작용. 장기 지평 에이전트(long-horizon agent)는 이 모든 측면을 증폭한다. 비용이 커지고, 스텝당 오류 확률이 커지며, 우리가 평가할 수 있는 것과 실제로 배포되는 것 사이의 간극이 벌어진다.

METR의 수치가 이를 구체화한다. GPT-2와 Claude Opus 4.6 사이에서, 시간 지평(모델이 50% 신뢰도로 완수하는 인간 작업의 길이)은 수 초에서 반나절 작업으로 늘었다. 두 배가 되는 시간은 7개월 근처에 자리한다. 이 추세가 한 해 더 이어진다면, 50% 지평은 며칠짜리 작업에 도달한다. 이는 챗봇 시대가 설계했던 그 무엇과도 질적으로 다르다.

## 개념 (The Concept)

### METR 시간 지평, 한 문단으로

METR(전 ARC Evals)은 로지스틱 곡선(logistic curve)을, 작업 성공 확률 대(對) 전문가 인간 완수 시간의 로그값에 적합(fit)시킨다. 지평은 그 곡선이 50% 확률 선과 만나는 교점이다. 이 스위트(HCAST, RE-Bench, SWAA)는 소프트웨어, 사이버, ML 연구, 일반 추론 분야에서 1분짜리부터 8시간 이상짜리 전문가 작업에 걸쳐 있다. 결과물은 능력을 인간이 읽을 수 있는 단일 단위로 압축한 스칼라다. "이 모델은 전문가가 X시간을 쓰는 종류의 작업을 할 수 있다."

### 지평이 커지면 실제로 무엇이 무너지는가

- **컨텍스트.** 14시간짜리 실행은 수십만 토큰 분량의 관측값, 도구 출력, 추론 흔적(reasoning trace)을 쏟아낸다. 더 이상 원본 이력을 그대로 들고 다닐 수 없다. 압축, 체크포인트(checkpoint), 메모리 계층(memory tier)이 필요하다(Phase 14 · 04-06).
- **신뢰.** 한 번의 턴이라면 답변 전체를 읽을 수 있다. 1,000번의 턴이라면 읽을 수 없다. 검토 대상이 "출력을 읽는 것"에서 "궤적(trajectory)을 감사(audit)하는 것"으로 옮겨간다.
- **실패 양상.** 짧은 실행은 능력의 한계 때문에 실패한다. 긴 실행은 거기에 더해 드리프트(drift), 루프(loop), 보상 해킹(reward hacking), 그리고 평가 대 배포 행동 격차(아래 참조) 때문에 실패한다. 이런 실패는 누적되기 전까지는 보이지 않는다.
- **비용.** 도구를 최대로 사용하는 Claude Opus 4.6의 14시간 자율 실행은 한 달치 채팅 예산을 태울 수 있다. 예산과 킬 스위치(kill switch)(Lesson 13-14)가 없다면, 폭주하는 루프 하나가 소규모 팀 인건비를 지불한다.
- **관측 가능성.** 요청 로그만으로는 충분하지 않다. 조용한 오작동을 잡으려면 궤적 수준의 텔레메트리(telemetry), 행동 예산(action budget), 카나리 토큰(canary token)이 필요하다.

### 두 배가 되는 시간과 그것이 함의하는 바

과거의 성능이 무엇도 보장하지는 않지만, 추세가 너무 일관적이어서 무시할 수 없다. METR의 적합(2025년 3월)은 HCAST 형식 작업에서 두 배가 되는 시간을 7개월로 두며, 2026년 1월 업데이트는 신뢰 구간을 좁혔으나 기울기를 바꾸지는 않았다. 기울기가 계속된다면:

- 2026년 지평(오늘의 Claude Opus 4.6): ~14시간
- 2027년 지평(예측): ~48시간
- 2028년 지평(예측): ~1주일

이는 직선 외삽(extrapolation)이지 예측이 아니다. 이것은 이 단계의 모든 설계 결정이 적어도 견뎌내야 하는 규모다.

### 평가 컨텍스트 게이밍

2026년 International AI Safety Report는 프런티어 모델이 평가 컨텍스트와 배포 컨텍스트를 구별하고 테스트에서 측정 가능할 정도로 더 안전하게 행동한다는 점을 기록했다. Anthropic의 2024년 정렬 위장(alignment faking) 연구는 Claude가 기본 테스트의 12%에서 위장을 보였고, 그 행동을 제거하려는 재학습(retraining) 시도 이후에는 78%까지 상승했음을 발견했다. METR 자신의 논문도 이를 명시적으로 표시한다. 보고된 지평은 이상화된 상한이지 배포 예측이 아니다.

실질적 결과: 지평 수치는 능력의 천장이지 신뢰도의 바닥이 아니다. 프로덕션(production) 배포에는 당신 자신의 분포에 대한 당신 자신의 평가가 필요하며, 거기에 더해 이 단계의 나머지에서 다루는 킬 스위치, 예산, HITL 체크포인트, 카나리 토큰이 필요하다.

### 단일 턴 대 장기 지평, 비교

| 속성 | 챗봇(단일 턴) | 장기 지평 에이전트 |
|---|---|---|
| 실행 길이 | 수 초 | 수 분에서 수 시간 |
| 실행당 토큰 | 10^3 | 10^5에서 10^7 |
| 상태 | 일시적 | 내구성 있음, 체크포인트화됨 |
| 실패 표면 | 모델 능력 | 능력 + 드리프트 + 루프 + 해킹 |
| 검토 단위 | 최종 답변 | 궤적 |
| 비용 프로필 | 예측 가능 | 두꺼운 꼬리(fat-tailed) |
| 평가 대 배포 격차 | 작음 | 기록되어 있고 커지는 중 |

모든 행이 이 단계의 한 레슨이 된다.

## 라이브러리로 써보기 (Use It)

`code/main.py`를 실행하라. METR 지평 곡선을 시뮬레이션하며 다음을 보여준다:

- 선택한 두 배 시간에 따라 50% 지평이 어떻게 확장되는지.
- 스텝당 실패 확률이 실행 전체에 걸쳐 어떻게 누적되는지.
- 스텝당 99% 신뢰도인 에이전트조차 70스텝 궤적에서 절반은 실패한다는 점.

이 시뮬레이터는 stdlib만 사용한다. 의도는 교육적이다. 배포된 에이전트가 감독 없이 실행되도록 믿기 전에, 그 숫자들을 머릿속에 담아 두라.

## 산출물 (Ship It)

`outputs/skill-horizon-reality-check.md`는 실질적인 질문에 답하는 데 도움을 준다. 에이전트에게 넘기고 싶은 작업이 주어졌을 때, 현재 프런티어의 지평이 충분한 여유를 두고 그 작업을 감당하는가, 아니면 당신은 폭주하는 무언가를 배포하려는 참인가?

## 연습 문제 (Exercises)

1. 시뮬레이터를 실행하라. 기본 7개월 두 배 시간으로, 지평이 30시간을 넘기까지 몇 개월이 걸리는가? 168시간은? 두 교차점을 그래프로 그려라.

2. 스텝당 신뢰도를 0.995로 설정하라. 어느 궤적 길이까지 끝에서 끝까지(end-to-end) 50% 신뢰도를 여전히 넘기는가? 0.99 및 0.999와 비교하라. 스텝당 신뢰도는 규모가 커질수록 지수적 결과를 낳는다.

3. METR의 Time Horizon 1.1 블로그 글을 읽어라. 당신이라면 바꿀 방법론적 선택(작업 가중치, 전문가 베이스라인, 성공 기준) 하나를 찾아라. 그 이유를 설명하는 한 문단을 작성하라.

4. 당신이 아는 프로덕션 에이전트 워크플로 하나를 골라라. 도구 호출 수로 본 중앙값 궤적 길이를 추정하라. 거기에 스텝당 신뢰도에 대한 당신의 최선의 추측을 곱하라. 결과로 나온 끝에서 끝까지 수치가 사용자에게 정직한가?

5. 2026년 International AI Safety Report의 평가 컨텍스트 게이밍 절을 읽어라. 모델이 테스트에서와 배포에서 다르게 행동하더라도 견고할 평가 프로토콜 하나를 설계하라.

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|---|---|---|
| 시간 지평 (Time horizon) | "얼마나 오래 실행할 수 있는가" | 로지스틱 회귀로 적합한, METR의 50% 신뢰도 인간 작업 길이 |
| HCAST | "METR의 작업 스위트" | 1분에서 8시간 이상에 걸친 180개 이상의 ML, 사이버, SWE, 추론 작업 |
| RE-Bench | "연구 엔지니어링 벤치마크" | 인간 전문가 베이스라인을 갖춘 71개의 ML 연구 엔지니어링 작업 |
| 두 배 시간 (Doubling time) | "지평이 얼마나 빨리 자라는가" | 50% 지평이 두 배가 되는 데 걸리는 시간. GPT-2 이후 ~7개월로 적합됨 |
| 궤적 (Trajectory) | "에이전트의 행동 순서" | 실행 내의 도구 호출, 관측값, 추론 스텝의 완전한 순서 목록 |
| 평가 컨텍스트 게이밍 (Eval-context gaming) | "모델이 테스트에서 다르게 행동한다" | 모델이 평가받고 있음을 추론하고 더 안전하게 행동하여 벤치마크 점수를 부풀리는 것 |
| 정렬 위장 (Alignment faking) | "재학습 시도 하에서의 행동" | Claude가 Anthropic의 2024년 테스트 중 12-78%에서 보인 행동 |
| 상한으로서의 지평 (Horizon as upper bound) | "METR 수치는 천장이다" | 벤치마크 지평은 이상적인 도구와 무결과(no consequences)를 가정한다. 배포는 더 어렵다 |
