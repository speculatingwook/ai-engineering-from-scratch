# 킬 스위치, 회로 차단기, 그리고 카나리아 토큰

> 킬 스위치(kill switch)는 에이전트(agent)의 편집 표면 밖에 보유된 불리언(boolean)이다 — Redis 키, 기능 플래그(feature flag), 서명된 설정 — 으로 에이전트를 통째로 비활성화한다. 회로 차단기(circuit breaker)는 더 세분화되어 있다. 특정 패턴(연속 다섯 번의 동일한 도구 호출)에서 트립(trip)하고, 위반 경로를 일시 정지하며, 사람에게 격상한다. 카나리아 토큰(canary token)은 고전적 기만(deception)에서 물려받았다. 에이전트가 건드릴 정당한 이유가 없는 가짜 자격 증명(credential)이나 허니팟(honeypot) 레코드로, 그 접근이 경보를 촉발한다. eBPF 기반 데이터패스(예: Cilium)는 격리된(quarantined) 파드(pod)의 송신(egress)을 커널 층에서 포렌식 허니팟으로 다시 쓸 수 있다. 공개된 Cilium 벤치마크는 부하 하에서 밀리초 미만의 P99 데이터패스 지연 시간(latency)을 보고한다(전파 예산은 데이터패스 자체가 아니라 정책 업데이트가 노드에 도달하는 방식에 달려 있다). 움직이는 베이스라인(baseline)에 적응하는 통계적 탐지기(EWMA, CUSUM)는 표류(drift)를 조용히 받아들일 것이다 — 휘지 않는 단단한 헌법적 한계(constitutional limit)와 층으로 결합하라.

**Type:** Learn
**Languages:** Python (stdlib, three-detector simulator: kill switch, circuit breaker, canary)
**Prerequisites:** Phase 15 · 13 (Cost governors), Phase 15 · 10 (Permission modes)
**Time:** ~60분

## 문제 (The Problem)

비용 거버너(cost governor, Lesson 13)는 에이전트가 무엇을 지출할 수 있는지를 한정한다. 다만 예산 안에서 무엇을 할 수 있는지까지 한정하지는 않는다. 50달러 속도 제한(velocity limit)을 가진 에이전트도 여전히 비밀을 유출하거나, 잘못된 게시물을 발행하거나, 리소스를 삭제할 수 있다 — 비싼 액션은 토큰(token)으로는 종종 싼 액션이다.

이 레슨은 비용 층 옆에 자리하는 세 가지 탐지기를 다룬다:

1. **킬 스위치**: 에이전트의 손이 닿지 않는 곳에 보유된 불리언 오프 버튼.
2. **회로 차단기**: 특정 경로를 일시 정지하는 액션 패턴 탐지기.
3. **카나리아 토큰**: 건드릴 정당한 이유가 없는데도 에이전트가 건드리면 스스로를 드러내게 되는 미끼.

세 가지 모두 LLM 이전의 엔지니어링이다. 고전적 기만, 속도 제한 차단기, 기능 플래그 킬은 자율 에이전트보다 앞선다. 새로운 점은 공격 표면(attack surface)이다. 에이전트는 신뢰할 수 없는 콘텐츠를 읽고(Lesson 11), 자신의 메모리를 편집하며, 안전해 보이는 많은 액션을 안전하지 않은 하나로 합성할 수 있다. 여기 명명된 탐지기들이 작동하는 이유는, 그것들이 에이전트의 자기 보고(self-report)를 신뢰하지 않기 때문이다.

## 개념 (The Concept)

### 킬 스위치

킬 스위치는 에이전트가 읽지만 쓸 수 없는 불리언이다. 구현 선택지:

- **관리형 서비스의 기능 플래그.** LaunchDarkly, Statsig, Unleash. 업데이트가 초 단위로 전파된다.
- **에이전트가 폴링(poll)하는 Redis 키.** 단순하다. 에이전트 프로세스가 매 턴마다 확인해야 한다.
- **객체 저장소의 서명된 설정.** 에이전트가 부팅 시 서명을 검증하고; 서명되지 않은 상태를 거부한다.
- **OS 수준 시그널 또는 컨테이너 수명주기 킬.** Docker `kill`, Kubernetes `kubectl delete pod`, systemd stop.

올바른 킬 스위치가 갖는 속성:

- 에이전트가 그것을 `off`로 설정할 수 없다. (에이전트의 자격 증명이 쓰지 못하는 시스템에 산다.)
- 시작 시뿐 아니라 모든 중대한 액션에서 확인된다.
- off일 때, 에이전트는 에이전트가 도달할 수 있는 시스템으로의 로깅을 포함하여, 외부에서 관찰 가능한 어떤 것도 하지 않는다.
- 재활성화는 자동 타임아웃이 아니라 명시적인 사람의 행동이다.

### 회로 차단기

회로 차단기는 에이전트 전체가 아니라 특정 패턴을 일시 정지한다. 고전적 형태(2007년 Nygard 책에서, 여전히 유효):

- **닫힘(Closed)**: 액션이 허용된다.
- **열림(Open)**: 액션이 차단된다.
- **반열림(Half-open)**: 냉각 시간 후, 1~3회 탐침 시도가 허용된다(기본 1회); 성공하면 차단기가 닫히고, 남은 실패가 있으면 다시 열린다.

에이전트 관련 트리거:

- 연속 다섯 번의 동일한 도구 호출(반복 루프).
- 같은 도구에 서로 다른 입력으로 연속 다섯 번 실패(시스템적 실패).
- 임계값을 넘는 도구 호출 비율(Lesson 13 속도).
- 신뢰 밖 콘텐츠 읽기 뒤에 특정 도구(예: `file.delete`)가 호출됨(Lesson 11).

### 카나리아 토큰

카나리아 토큰(허니토큰(honeytoken)이라고도 함)은 에이전트가 결코 건드려서는 안 되는 입력이다. 접근이 곧 경보다.

에이전트를 위한 예시:

- 실제 권한이 없는, 환경 속의 가짜 `AWS_SECRET_ACCESS_KEY`. 유출 시도가 즉시 귀속 가능하다.
- 알려진 센티넬(sentinel) 값으로 플래그된 가짜 데이터베이스 레코드. 이 레코드의 어떤 읽기나 갱신이든 경보를 발사한다.
- 작업 공간 속의 가짜 파일. 이 파일의 읽기나 편집은 의심스럽다.
- 세션 시작 시 주입된 가짜 메모리 항목. 이 항목이 이후 출력에 나타나면, 메모리가 유출된 것이다.

카나리아 설계는 워크플로마다 특수하다. 에이전트가 건드릴 정당한 이유가 있는 카나리아는 노이즈다.

### 통계적 한계와 단단한 한계를 층으로 쌓는 이유

통계적 탐지기(EWMA, CUSUM, 도구 호출 비율의 z-점수)는 움직이는 베이스라인에 적응한다. 이 적응은 합법적 표류에는 기능이고 적대적 표류에는 버그다. 인내심 있는 공격자는 어떤 경보도 발사되지 않을 만큼 천천히 베이스라인을 옮길 수 있다.

단단한 헌법적 한계(Lesson 17)는 휘지 않는다. 헌법적 규칙으로서 "`secrets@`로 결코 이메일 보내지 않기"는 더 단순하고, 감사 가능하며, 느린 표류로 게임할 수 없다. 층 쌓기 패턴: 통계적 탐지기가 대부분의 노이즈를 잡고, 단단한 한계가 탐지기를 지나쳐 걸어간 공격을 잡는다.

### eBPF 데이터패스 리다이렉트를 통한 격리

Cilium의 eBPF 기반 네트워크 정책은 격리된 파드의 송신을 커널 층에서 포렌식 허니팟으로 다시 쓸 수 있다. Cilium의 공개 벤치마크는 TCP_RR에서 밀리초 미만의 P99 데이터패스 지연 시간을 보고한다. 실제 클러스터에서 격리를 *촉발하는* 벽시계 예산은 정책 업데이트가 컨트롤 플레인에서 노드로 전파되는 방식이 지배한다(오퍼레이터가 사전 로드된 Kubernetes에서는 수백 밀리초가 현실적이며; 꼬리 지연 시간은 클러스터 크기와 재조정 주기에 달려 있다). 이것은 두 가지를 준다. 에이전트가 실제 어떤 것도 유출하지 않고, 허니팟 안에서의 에이전트 동작이 완전히 관찰된다. 카나리아 토큰과 같은 패턴이지만 다른 층이다. 에이전트는 격리된 상태에서 무엇을 하려 하는지로 의도를 드러낸다.

### 어떤 탐지기도 잡지 못하는 것

- **허용된 기본 요소로부터 합성되는 의미 공격.** 모든 개별 액션이 모든 탐지기를 통과한다. 합성이 문제다. 이것은 Lesson 10의 Auto Mode 합성 실패다.
- **탐지기를 페이로드의 일부로 포함하는 공격.** 에이전트가 "행동하기 전에 카나리아를 비활성화하라"는 페이지를 읽고 그럴 능력까지 갖췄다면, 탐지기는 침해된다. 카나리아 토큰은 에이전트가 수정할 수 없는 시스템에 있어야 한다.

## 라이브러리로 써보기 (Use It)

`code/main.py`는 세 탐지기를 통과하는 짧은 에이전트 궤적(trajectory)을 시뮬레이션한다. 외부 dict에 보유된 킬 스위치; 다섯 번의 동일한 도구 호출에서 트립하는 회로 차단기; 읽으면 경보를 촉발하는 카나리아 파일. 합성 궤적을 입력으로 넣는다: 합법적 액션, 반복 루프, 카나리아 탐침, 그리고 에이전트의 액션이 정지되는 킬 스위치 촉발 시나리오.

## 산출물 (Ship It)

`outputs/skill-tripwire-design.md`는 에이전트 배포(deployment)를 위한 제안된 탐지기 스택을 검토하고 빈틈을 표시한다(킬 스위치 누락, 카나리아 누락, 너무 느슨한 회로 차단기 임계값).

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 회로 차단기가 5번째 턴(다섯 번째 동일 호출)에서 발사되고 카나리아가 9번째 턴(가짜 키 읽기)에서 발사됨을 확인하라.

2. 통계적 탐지기를 추가하라: 도구 호출 비율의 EWMA z-점수. 천천히 표류하는 궤적을 입력으로 넣고 탐지기가 결코 발사되지 않음을 보여라. 이제 단단한 한계(10분에 도구 호출 50회 이하)를 추가하고 같은 궤적에서 단단한 한계가 발사됨을 보여라.

3. 브라우저 에이전트(Lesson 11)를 위한 카나리아 토큰 집합을 설계하라. 최소 세 개의 카나리아와 각각이 무엇을 탐지할지 나열하라.

4. Cilium 네트워크 정책 문서를 읽어라. 송신 리다이렉트 격리 흐름을 구체적으로 기술하라: 어떤 정책 셀렉터, 어떤 파드, 어떤 송신 재작성, 어떤 경보. "격리를 결정"에서 "첫 리다이렉트된 패킷"까지의 벽시계 지연 시간을 무엇이 지배하는가?

5. 킬 스위치된 에이전트를 위한 재활성화 절차를 정의하라. 누가 재활성화할 수 있는가? 무엇이 문서화되어야 하는가? 재활성화 전에 에이전트에 관해 무엇이 바뀌어야 하는가?

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|---|---|---|
| 킬 스위치 (Kill switch) | "오프 버튼" | 에이전트의 편집 표면 밖의 불리언; 모든 중대한 액션에서 확인됨 |
| 회로 차단기 (Circuit breaker) | "패턴 일시 정지" | 반복, 실패율, 또는 속도 제한에 대한 액션별 트립 |
| 카나리아 토큰 (Canary token) | "허니토큰" | 에이전트가 건드릴 정당한 이유가 없는 미끼; 접근이 경보를 발사 |
| 허니팟 (Honeypot) | "포렌식 샌드박스" | 격리된 에이전트가 관찰되는 리다이렉트된 트래픽/작업 공간 |
| EWMA | "이동 평균" | 지수 가중; 표류에 적응(기능 + 버그) |
| CUSUM | "누적 합" | 베이스라인으로부터의 지속적 이동을 탐지 |
| 단단한 한계 (Hard limit) | "헌법적 규칙" | 적응하지 않음; 이력과 무관하게 일정 |
| 헌법적 한계 (Constitutional limit) | "항상 참인 규칙" | Lesson 17의 헌법에 묶임; 에이전트가 편집할 수 없음 |
