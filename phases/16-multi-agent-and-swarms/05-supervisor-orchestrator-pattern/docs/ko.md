# 슈퍼바이저 / 오케스트레이터-워커 패턴 (Supervisor / Orchestrator-Worker Pattern)

> 하나의 리드(lead) 에이전트(agent)가 계획하고 위임하며; 전문화된 워커(worker)들이 병렬 컨텍스트에서 실행하고 보고한다. 이것이 Anthropic의 Research 시스템(리드로서 Claude Opus 4, 서브에이전트(subagent)로서 Sonnet 4) 뒤에 있는 패턴으로, 내부 연구 평가에서 단일 에이전트 Opus 4 대비 +90.2%로 측정되었다. Anthropic의 엔지니어링 포스트는 BrowseComp에서의 분산(variance)의 80%가 토큰(token) 사용량만으로 설명된다고 보고한다 — 멀티 에이전트(multi-agent)가 이기는 것은 주로 각 서브에이전트가 신선한 컨텍스트 윈도우(context window)를 받기 때문이다. 이 레슨은 프리미티브(primitive)로부터 슈퍼바이저 패턴을 만들고, 프로덕션(production) 배포(deployment)에서 나온 2026년 엔지니어링 교훈을 다룬다.

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`)
**Prerequisites:** Phase 16 · 04 (Primitive Model)
**Time:** ~75분

## 문제 (Problem)

연구(research)는 단일 에이전트 시스템이 실패하는 전형적인 작업이다. "2023년과 2026년 사이에 멀티 에이전트 시스템에서 무엇이 바뀌었는가?"라고 물었다고 하자. 단일 에이전트는 다섯 편의 논문을 순차적으로 읽고 그 텍스트로 컨텍스트의 절반을 채운 다음, 그 모두를 함께 추론해야 한다. 다섯 번째에 도달할 즈음이면 첫 논문을 잊는다. 병렬화할 수 없다.

슈퍼바이저(supervisor) 패턴이 이를 고친다. 하나의 리드 에이전트가 검색을 계획하고 각 하위 질문을 워커에게 위임한 뒤 종합한다. 각 워커는 좁은 질문 하나를 위해 자신만의 200k 토큰 윈도우를 받는다. 리드는 원본 논문을 결코 보지 않는다 — 오직 워커 요약만 본다.

Anthropic의 프로덕션 Research 시스템은 단일 Opus 4 대비 내부 연구 평가에서 +90.2%를 보고한다. 같은 포스트는 BrowseComp 분산의 80%가 *토큰 사용량만으로* 설명된다고 언급한다. 서브에이전트당 신선한 컨텍스트가 주된 메커니즘이다.

## 개념 (Concept)

### 패턴 (The pattern)

```
                 ┌──────────────┐
                 │   Lead       │  plans, decomposes,
                 │  (Opus 4)    │  synthesizes
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         fresh       fresh        fresh
         context     context      context
```

리드는 원본 자료를 결코 읽지 않는다. 워커들은 리드가 종합하기 전까지 서로의 작업을 보지 못한다. 각 화살표는 좁은 아티팩트(artifact)를 가진 핸드오프(handoff)다.

### 왜 이기는가

세 가지 메커니즘:

1. **서브에이전트당 신선한 컨텍스트.** "FIPA-ACL 유산"을 탐색하는 워커는 리드가 계획에 쓴 40k 토큰을 지니지 않는다. 질문 하나를 위해 200k 윈도우를 받는다.
2. **프롬프트를 통한 전문화.** 리드의 프롬프트는 "분해하고 종합하라"이지 "연구하라"가 아니다. 각 워커의 프롬프트는 좁다: "X에서 무엇이 바뀌었는지 찾아라." 집중된 프롬프트는 집중된 출력을 낳는다.
3. **병렬성.** 워커들은 동시에 실행된다. 벽시계 시간(wall-clock time)은 대략 `sum(worker_times)`가 아니라 `max(worker_times) + plan + synthesis`다.

### 엔지니어링 교훈 (Anthropic 2025)

Anthropic 포스트는 2026년에도 여전히 유효한 여러 프로덕션 교훈을 나열한다.

- **질의 복잡도에 노력을 맞춰라.** 단순 질의: 에이전트 하나, 도구 호출 3-10회. 복잡한 질의: 10개 이상의 에이전트. 호출자가 아니라 리드가 이를 추정해야 한다.
- **넓게, 그다음 좁게.** 먼저 넓은 하위 질문으로 분해하고, 그다음 답이 깊이를 요구하면 하위 질문당 더 많은 워커를 생성하라.
- **레인보우 배포(Rainbow deployments).** 에이전트는 장시간 실행되고 상태를 가진다. 전통적인 블루-그린(blue-green)은 동작하지 않는다. Anthropic은 레인보우를 쓴다. 기존 버전이 빠지는 동안 새 버전을 점진적으로 롤아웃하는 방식이다.
- **토큰 사용량이 지배한다.** 멀티 에이전트는 단일 에이전트의 약 15배 토큰이다. 작업 가치가 비용을 정당화할 때만 실행하라.

### LangGraph의 전환

LangGraph는 원래 고수준 `create_supervisor` 헬퍼를 가진 `langgraph-supervisor` 라이브러리를 출시했다. 2025년에 LangChain은 권장 사항을 도구 호출(tool-calling)로 슈퍼바이저 패턴을 직접 구현하는 쪽으로 옮겼는데, 도구 호출이 *슈퍼바이저가 무엇을 보는지*를 더 많이 제어(컨텍스트 엔지니어링)하게 해주기 때문이다. 라이브러리는 여전히 동작하지만, 문서는 이제 도구 호출 형태를 권장한다.

### 실패 양식 (The failure modes)

- **리드가 계획을 환각(hallucinate)한다.** 리드가 실제 질문을 분해하지 않는 하위 질문을 생성하면, 워커들은 잘못된 대상에 대해 정밀한 연구를 한다.
- **워커가 과도하게 탐색한다.** 명시적 범위 경계 없이는, 워커들이 할당된 하위 질문을 넘어 표류하여 종합 단계를 오염시킨다.
- **종합 충돌.** 두 워커가 모순되는 사실을 반환한다. 리드는 다시 묻거나(라운드를 추가) 불일치를 명시적으로 기록해야 한다. 한쪽을 조용히 고르는 것이 최악의 실패다. 사용자는 불일치가 일어났음을 결코 알지 못한다.

### 슈퍼바이저가 틀린 때

- **순차적 작업.** 2단계가 문자 그대로 1단계의 출력을 필요로 한다면, 병렬성은 아무것도 얻지 못한다. 파이프라인(CrewAI Sequential, LangGraph 선형 그래프)을 사용하라.
- **단순 질의.** 단일 에이전트가 더 빠르고 저렴하게 처리한다. 워커를 생성하기 전에 리드의 "노력 맞춤" 점검을 사용하라.
- **엄격한 결정성.** 슈퍼바이저는 LLM 선택 위임을 사용한다. 감사/재생이 적응성보다 더 중요할 때는 정적 그래프가 낫다.

## 직접 만들기 (Build It)

`code/main.py`는 `threading`을 사용해 세 병렬 워커의 슈퍼바이저를 구현한다. 리드가 질의를 하위 질문으로 분해하고, 워커들이 각 하위 질문에서 동시에 실행되고, 리드가 종합한다. 실제 LLM은 없다 — 워커들은 페치-앤-요약(fetch-and-summarize)을 시뮬레이션하도록 스크립트되어 있다.

핵심 구조:

- `Lead.plan(query)`는 질의를 3개의 하위 질문으로 나눈다.
- `Worker.run(sub_q)`는 가짜 요약을 반환한다(프로덕션에서는 임의의 도구 사용 에이전트일 수 있다).
- `Lead.run(query)`는 스레드에서 워커를 시작하고, 조인(join)하고, 종합한다.

실행:

```
python3 code/main.py
```

출력은 계획, 시작/종료 타임스탬프가 있는 병렬 워커 트레이스(trace), 그리고 최종 종합을 보여준다. 벽시계 이득을 볼 수 있다: 세 개의 0.3초 워커가 0.9초가 아니라 약 0.35초에 실행된다.

## 라이브러리로 써보기 (Use It)

`outputs/skill-supervisor-designer.md`는 사용자 질의를 받아 슈퍼바이저 패턴 설계를 만들어낸다. 리드 시스템 프롬프트, 워커 역할, 하위 질문 분해 규칙, 종합 템플릿이 그 결과물이다. 새 연구 스타일 에이전트 시스템을 만들기 전에 이를 사용하라.

## 산출물 (Ship It)

슈퍼바이저 패턴을 배포하기 전 체크리스트:

- **모델 짝짓기.** 리드는 추론 등급 모델(Opus 급, `o3` 급)에. 워커는 더 빠르고 저렴한 모델(Sonnet, `o4-mini`)에.
- **워커 타임아웃.** 중앙값 런타임의 2배를 초과하는 워커는 종료된다; 리드는 더 좁은 범위로 다시 생성하거나 그것 없이 진행한다.
- **워커당 토큰 상한.** 하드 제한(가령 예상 종합 입력의 10배)이 폭주하는 워커가 예산을 날리는 것을 막는다.
- **관측 가능성(observability).** 리드의 계획, 각 워커의 도구 호출, 그리고 종합을 추적하라. 이것이 사후 디버깅의 기반이다.
- **레인보우 롤아웃.** 상태를 가진 장시간 실행 에이전트는 핫 스왑(hot swap)이 아니라 점진적 버전 전환이 필요하다.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행한 다음, 리드가 3개 대신 5개의 워커를 생성하도록 수정하라. 벽시계 효과를 관찰하라. 이 데모에서 워커 수가 몇 개일 때 생성 오버헤드가 병렬 절감을 초과하는가?
2. 워커 타임아웃을 구현하라: 0.5초보다 오래 실행되는 워커를 종료하고 리드가 남은 결과를 종합하게 하라. 워커가 잘렸음을 알기 위해 어떤 관측 가능성이 필요한가?
3. 리드의 종합에 충돌 탐지 단계를 추가하라: 두 워커가 모순되는 답을 반환하면, 리드가 하나를 고르는 대신 불일치를 기록한다. LLM을 호출하지 않고 모순을 어떻게 탐지하는가?
4. Anthropic의 Research 시스템 엔지니어링 포스트를 읽어라. 이 장난감 데모가 프로덕션에서 실행되려면 채택해야 할 세 가지 관행을 나열하라.
5. LangGraph의 `create_supervisor`(레거시)와 새로운 도구 호출 권장 사항을 비교하라. 어느 것이 슈퍼바이저가 무엇을 보는지에 대해 더 나은 제어를 주는가? Anthropic은 왜 종합에 원본 워커 컨텍스트가 아니라 하위 답변만 명시적으로 전달하는가?

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|------|----------------|------------------------|
| 슈퍼바이저 (Supervisor) | "리드 에이전트" | 계획하고, 위임하고, 종합하는 오케스트레이터 에이전트. 일 자체는 하지 않는다. |
| 워커 (Worker) | "서브에이전트" | 좁은 범위와 자신의 컨텍스트 윈도우를 가지고 슈퍼바이저가 호출하는 집중된 에이전트. |
| 오케스트레이터-워커 (Orchestrator-worker) | "슈퍼바이저 패턴" | 같은 것, 다른 이름. 2026년 문헌은 둘 다 사용한다. |
| 신선한 컨텍스트 (Fresh context) | "깨끗한 윈도우" | 워커의 컨텍스트는 리드의 이력이 아니라 자신의 시스템 프롬프트와 할당된 질문에서 시작한다. |
| 레인보우 배포 (Rainbow deployment) | "점진적 롤아웃" | 장시간 실행 상태 보존 에이전트는 블루-그린이 아니라 버전이 매겨진 드레인-앤-교체가 필요하다. |
| 토큰 지배 (Token dominance) | "컨텍스트가 변수다" | Anthropic에 따르면 연구 평가 분산의 80%가 모델 선택이 아니라 사용된 총 토큰에서 온다. |
| 노력 맞춤 (Scale effort) | "에이전트 수를 복잡도에 맞춰라" | 리드가 질의 난이도를 추정하고, 그에 맞춰 1개 대 10개 이상의 워커를 생성한다. |
| 종합 충돌 (Synthesis conflict) | "워커들이 불일치한다" | 두 워커가 모순되는 사실을 반환한다; 리드는 조용히 하나를 고르지 말고 불일치를 드러내야 한다. |
