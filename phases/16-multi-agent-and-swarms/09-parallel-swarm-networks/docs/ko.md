# 병렬 / 스웜 / 네트워크형 아키텍처 (Parallel / Swarm / Networked Architectures)

> 슈퍼바이저(supervisor)와의 대조: 중앙 결정자가 없다. 에이전트(agent)들이 공유 이벤트 버스(event bus)를 읽고, 비동기적으로 작업을 가져오고, 결과를 다시 쓴다. LangGraph는 탈중앙화되고 동적인 환경을 위한 "Swarm Architecture"를 명시적으로 지원한다. Matrix(arXiv:2511.21686)는 제어 흐름과 데이터 흐름 모두를 분산 큐(queue)를 통해 전달되는 직렬화된 메시지로 표현하여 오케스트레이터(orchestrator) 병목을 제거한다. 트레이드오프(trade-off)는 명시적이다: 확장성을 위한 결정성과 추적 가능성의 교환. 스웜(swarm)은 많은 독립적 하위 문제를 가진 작업에 맞는다; 단일한 일관된 계획이 필요한 작업에는 맞지 않는다.

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`, `queue`)
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern), Phase 16 · 04 (Primitive Model)
**Time:** ~75분

## 문제 (Problem)

슈퍼바이저는 소수의 워커(worker)로 확장된다. 수백 개라면 어떨까? 슈퍼바이저 자체가 병목이 된다: 누가 무엇을 하는지에 대한 모든 결정이 한 에이전트를 통해 깔때기처럼 모인다. 한 느린 계획 단계가 전체 시스템을 멈춘다.

스웜 아키텍처는 설계를 뒤집는다. 중앙 플래너(planner)가 작업을 디스패치하는 대신, 워커들이 공유 큐에서 작업을 가져온다. "조율"은 이벤트 버스 의미론에 구워져 있다. 오케스트레이터가 없다; 시스템은 큐가 확장되는 만큼 확장된다.

## 개념 (Concept)

### 형태 (The shape)

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

오케스트레이터가 없다. 각 워커는 반복한다: 작업을 가져오고, 처리하고, 결과를 쓴다(그리고 선택적으로 후속 작업을 큐에 넣는다).

### 스웜이 맞을 때

- **많은 독립적 작업.** 스크래핑, 변환, 분류. 작업들이 서로 의존하지 않는다.
- **가변 지속 시간 작업.** 일부 작업이 100ms 걸리고 다른 것들이 10초 걸린다면, 스웜은 부하를 자동으로 균형 잡는다 — 빠른 워커가 다음 작업을 가져온다. 슈퍼바이저는 지속 시간을 예측해야 한다.
- **결정성보다 처리량(throughput).** 엄격한 순서가 아니라 총 완료 시간에 신경 쓴다.

### 스웜이 실패할 때

- **순서가 있는 워크플로.** 3단계가 2단계의 출력을 필요로 한다면, 스웜은 2단계가 끝나기 전에 3단계가 발동될 위험이 있다.
- **전역 계획 작업.** 복잡한 연구 질문은 플래너의 이득을 본다. 연구자 스웜은 일관된 보고서가 아니라 독립적 사실들을 생산한다.
- **디버깅.** 중앙 로그가 없고 비동기 작업이 있으면, 버그를 재현하는 것이 비싸다.

### Matrix (arXiv:2511.21686)

Matrix는 스웜을 자연스러운 결론까지 밀어붙인 2025년 논문이다: 제어 흐름과 데이터 흐름 모두가 분산 큐 위의 직렬화된 메시지다. 중앙 코디네이터가 없다. 결함 내성(fault tolerance)은 메시지 내구성(durability)에서 온다. 확장성은 시스템이 아니라 메시지 브로커(broker)의 문제다.

기여: 멀티 에이전트 조율이 "슈퍼바이저가 다음에 어떤 에이전트를 고르는가?"가 아니라 "이 에이전트는 어떤 메시지 토픽(topic)을 구독하는가?"인 프로그래밍 모델. 이는 시스템을 pub/sub 이벤트 메시(mesh)처럼 보이게 한다.

### LangGraph의 Swarm Architecture

LangGraph 2025 문서는 "Swarm Architecture"를 멀티 에이전트 패턴 중 하나로 명시적으로 기술한다: 에이전트가 노드지만, 간선(edge)이 사이클(cycle)을 가진 방향 그래프를 형성하고 어떤 노드든 풀에서 활성화될 수 있다. 워커는 슈퍼바이저 할당이 아니라 조건에 따라 가용한 작업에서 가져온다.

### 실패 양식: 기아와 핫스팟 (starvation and hot-spotting)

모든 워커가 가장 빨리 가용한 작업을 가져오면, 장시간 실행 작업은 그것만 남을 때까지 결코 선택되지 않는다. 고전적 큐 기아(starvation).

완화책:
- 명시적 에이징(aging)을 가진 우선순위 큐(대기 시간에 따라 우선순위 증가).
- 워커 전문화: 일부 워커는 "긴" 작업만 받는다.
- 백프레셔(back-pressure): 얼마나 많은 빠른 작업이 큐에 들어가는지 제한.

### 콘텐츠 기반 라우팅 연결고리

스웜은 콘텐츠 기반 라우팅(content-based routing)(Lesson 22)과 자연스럽게 짝을 이룬다. 범용 큐 대신, 메시지 타입당 하나의 큐를 둔다. 전문가 워커는 자기 타입만 구독한다. 이것이 수천 개의 에이전트로 확장되는 메시지 버스 아키텍처의 기반이다.

## 직접 만들기 (Build It)

`code/main.py`는 공유 `queue.Queue`에서 가져오는 4개의 워커 스레드 스웜을 구현한다. 작업은 가변 지속 시간을 가진다(일부 빠르고, 일부 느림). 데모는 다음을 대조한다.

- **순차 베이스라인(baseline):** 하나의 워커가 모든 작업을 직렬로 처리한다.
- **고정 할당:** 각 작업이 특정 워커에 사전 할당됨(슈퍼바이저 스타일).
- **스웜:** 워커들이 공유 큐에서 가져온다.

스웜은 부하를 자동으로 균형 잡는다; 고정 할당은 할당된 작업이 느릴 때 빠른 워커를 놀게 둔다.

실행:

```
python3 code/main.py
```

출력은 워커별 작업 수(스웜은 고르지 않지만 최적으로 분배한다)와 벽시계 시간(wall-clock time)을 보여준다.

## 라이브러리로 써보기 (Use It)

`outputs/skill-swarm-fit.md`는 작업이 스웜을 써야 하는지 슈퍼바이저를 써야 하는지 평가한다. 입력: 작업 독립성, 지속 시간 분산(variance), 순서 요구사항, 디버깅 가능성 필요.

## 산출물 (Ship It)

체크리스트:

- **에이징을 가진 우선순위 큐.** 장기 작업 기아를 막는다.
- **워커 멱등성(idempotency).** 워커가 실행 도중 충돌하면 작업이 두 번 이상 가져와질 수 있다. 워커는 멱등이어야 한다.
- **내구성 있는 큐.** 프로덕션(production)에는 Kafka, Redis Streams, 또는 데이터베이스 기반 큐를 사용하라. `queue.Queue`는 메모리 전용이다.
- **작업별 관측 가능성(observability).** 모든 작업은 추적 ID를 가진다; 모든 워커는 그것과 함께 시작/종료를 로깅한다.
- **백프레셔.** 큐가 워커가 비우는 것보다 빠르게 커지면, 생산자를 늦춰라.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 가변 지속 시간 워크로드에서 스웜은 순차보다 얼마나 빠른가? 고정 할당보다 얼마나 빠른가?
2. 우선순위 큐 변형을 추가하라(`queue.PriorityQueue` 사용). 작업 "importance" 필드로 우선순위를 할당하라. 지속적 부하에서 낮은 우선순위 작업이 기아에 빠지는지 관찰하라.
3. 핫스팟 탐지기를 구현하라: 어떤 워커가 가장 느린 워커보다 3배 더 많은 작업을 처리할 때 로깅하라. 그것은 작업 지속 시간 분포에 대해 무엇을 가리키는가?
4. Matrix 논문(arXiv:2511.21686) 초록과 섹션 3을 읽어라. Matrix가 받아들이는 구체적 트레이드오프 하나(확장성 이득)와 포기하는 하나(추적 가능성, 결정성)를 식별하라.
5. 스웜 데모를 (task_type, payload) 튜플의 `queue.Queue`를 사용하도록 변환하고, 워커들이 특정 타입만 구독하게 하라. 작업이 이질적일 때 어떤 라우팅 규칙이 타당한가?

## 핵심 용어 (Key Terms)

| 용어 | 흔히 하는 말 | 실제 의미 |
|------|----------------|------------------------|
| 스웜 아키텍처 (Swarm architecture) | "탈중앙화된 에이전트" | 워커들이 공유 큐에서 가져온다; 중앙 오케스트레이터가 없다. |
| 이벤트 버스 (Event bus) | "에이전트가 토픽을 구독한다" | 타입이나 콘텐츠로 작업을 워커에게 라우팅하는 메시지 브로커. |
| 기아 (Starvation) | "작업이 결코 실행되지 않는다" | 더 높은 우선순위 작업이 지속적으로 도착하기 때문에 낮은 우선순위 작업이 결코 선택되지 않음. |
| 핫스팟 (Hot-spotting) | "한 워커가 익사한다" | 한 워커가 대부분의 작업을 받는 부하 불균형. |
| 백프레셔 (Back-pressure) | "생산자를 늦춰라" | 큐가 가득 찰 때 상류에 생산 중단을 신호하는 메커니즘. |
| 멱등 워커 (Idempotent worker) | "재실행해도 안전" | 두 번 처리된 작업이 같은 결과를 낳는다. 워커가 실행 도중 충돌할 수 있으므로 필요하다. |
| 내구성 있는 큐 (Durable queue) | "충돌에서 살아남는다" | 디스크나 복제된 저장소가 받쳐주는 큐; 워커가 충돌해도 작업이 손실되지 않는다. |
| Matrix 프레임워크 (Matrix framework) | "완전한 메시지 전달 스웜" | 데이터와 제어 흐름 모두가 분산 큐 위의 직렬화된 메시지다. |
