# 실패 모드 — MAST, 집단사고, 단일 문화, 연쇄 오류 (Failure Modes — MAST, Groupthink, Monoculture, Cascading Errors)

> 2026년의 참조 분류 체계는 **MAST** (Cemri et al., NeurIPS 2025, arXiv:2503.13657)이며, 이는 **41–86.7% 실패율**을 보이는 7개의 최첨단 오픈소스 MAS에 걸친 1642개의 실행 추적에서 도출되었다. 세 가지 근본 범주: **명세 문제(Specification Problems)**(41.77%) — 역할 모호성, 불명확한 과제 정의; **협응 실패(Coordination Failures)**(36.94%) — 통신 붕괴, 상태 비동기화; **검증 격차(Verification Gaps)**(21.30%) — 누락된 검증, 부재한 품질 검사. **집단사고(Groupthink)** 계열(arXiv:2508.05687)은 다음을 더한다: 단일 문화 붕괴(같은 기반 모델 → 상관된 실패), 동조 편향(에이전트(agent)들이 서로의 오류를 강화), 결함 있는 마음 이론, 혼합 동기 동역학, 연쇄 신뢰성 실패. 연쇄 예시: 지불 실패가 주문 재시도를 유발하고, 그것이 재고 재시도를 유발하며, 그것이 재고 서비스를 압도하는 재시도 폭풍(retry storm)(초 단위로 10배 부하 — 회로 차단기(circuit breaker)가 필요). 메모리 오염(memory poisoning): 한 에이전트의 환각(hallucination)이 공유 메모리에 들어가고, 하류 에이전트들이 그것을 사실로 취급한다. 정확도가 점진적으로 감쇠하여 근본 원인 진단을 고통스럽게 만든다. **STRATUS** (NeurIPS 2025)는 특화된 탐지 / 진단 / 검증 에이전트를 통해 완화 성공이 1.5배 개선됨을 보고한다. 이 레슨은 실패 모드를 일급 엔지니어링 대상으로 다룬다.

**Type:** Learn
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 13 (Shared Memory), Phase 16 · 14 (Consensus and BFT), Phase 16 · 15 (Voting and Debate Topology)
**Time:** ~75분

## 문제 (Problem)

다중 에이전트(multi-agent) 시스템은 실제 과제에서 41-86.7%의 비율로 실패한다(Cemri et al. 2025가 7개의 오픈소스 MAS에 걸쳐 이를 측정했다). 그것은 "그냥 에이전트를 더 추가하라"로 디버깅되지 않는다. 실패에는 구조적 원인이 있다. MAST 분류 체계는 당신에게 범주를 준다. 이 레슨은 각 범주를 구체적 탐지, 진단, 완화 패턴에 매핑하여 그 숫자들이 자의적으로 보이지 않게 한다.

2026년 프로덕션(production) 관행은 실패 모드를 설계 입력으로 다루는 것이다. 당신의 아키텍처는 각 MAST 범주를 짚어 당신이 배포한 완화책의 이름을 댈 수 있을 때까지는 "충분히 좋지" 않다.

## 개념 (Concept)

### MAST 범주

**명세 문제(전체 실패의 41.77%).** 에이전트의 과제가 충분히 빡빡하게 정의되지 않았다. 예시:

- 역할 모호성: 두 에이전트가 모두 자신이 검토자라고 생각한다.
- 과제 명세 부족: 사용자가 특정 관점을 원했는데 "이것을 요약하라"고만 함.
- 암묵적 성공 기준: 에이전트가 자신이 성공했는지 알 수 없음.

완화책:
- 명시적 역할 계약을 작성하라. 각 에이전트의 프롬프트(prompt)는 자신이 무엇을 하는지 *그리고 무엇을 하지 않는지*를 명시한다.
- 과제별 인수 테스트. 에이전트가 시작하기 전에 "완료는 X처럼 보인다"를 정의하라.
- 사전 점검 명세 검사: 별도 에이전트가 디스패치 전에 과제 정의를 검토한다.

**협응 실패(36.94%).** 통신 또는 상태 붕괴.

예시:
- 두 에이전트가 동기화 없이 공유 상태를 갱신한다.
- 에이전트 간 메시지 손실(큐(queue) 실패, 타임아웃).
- 상태 표류: 에이전트 A는 과제가 끝났다고 생각하고, 에이전트 B는 여전히 실행 중이다.

완화책:
- 낙관적 동시성을 갖춘 버전 관리되는 공유 상태.
- 중요한 메시지에 대한 명시적 확인(승인될 때까지 재시도).
- 주기적 상태 동기화 체크포인트. 표류를 일찍 탐지하라.

**검증 격차(21.30%).** 출력에 대한 독립적 검사 없음.

예시:
- 한 에이전트가 성공을 주장한다. 아무도 검증하지 않는다.
- 에이전트 사슬에서 각자가 이전 것의 출력을 신뢰한다.
- 창발적 합성 행동에 대한 테스트 커버리지 누락.

완화책:
- 독립적 검증자 에이전트(레슨 13). 읽기 전용, 독립적 출처 접근.
- 명시적 핸드오프 계약: "A의 출력은 B가 시작하기 전에 검사자 C를 통과해야 한다."
- 사후 분석을 위한 결과 로깅.

### 집단사고 계열 (arXiv:2508.05687)

에이전트들이 동질화되거나 서로를 모방할 때 발생하는 다섯 가지 관련 실패:

**단일 문화 붕괴.** 같은 기반 모델 또는 학습 데이터 → 상관된 오류. 세 에이전트가 LLM을 공유하면, 그들은 그것의 환각을 공유한다.

**동조 편향.** 에이전트들이 틀렸을 때조차 가장 목소리 크거나 가장 자신감 있는 동료 쪽으로 조정한다.

**결함 있는 ToM.** 에이전트들이 서로의 믿음을 모델링하는 데 실패한다. 협응이 무너진다(레슨 18).

**혼합 동기 동역학.** 부분적으로 정렬된 유인을 가진 에이전트들이 아무도 만족시키지 못하는 타협-중간으로 표류한다.

**연쇄 신뢰성 실패.** 한 구성요소의 오류 패턴이 의존 구성요소에서 오류 패턴을 유발한다.

### 연쇄 예시 — 재시도 폭풍

전형적인 2026년 사고 패턴:

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

해결책은 고전적이다: **회로 차단기**. 하류 오류율이 임계값을 초과하면, 캐시된 또는 기본 결과로 단락시킨다. 더불어 요청당 상한이 있는 재시도 예산.

회로 차단기는 분산 시스템에서 수정 없이 직접 빌려오는 몇 안 되는 다중 에이전트 실패 완화책 중 하나다.

### 메모리 오염 (재방문)

레슨 13에서: 한 에이전트의 환각이 공유 메모리 사실이 되고, 하류 에이전트들이 그 오염된 사실에 대해 추론한다. MAST 용어로, 이것은 공유 메모리 계층의 검증 격차다.

점진적 정확도 감쇠가 증상이다. 충돌을 얻는 게 아니라, 근본 원인을 찾기 어려운 느린 표류를 얻는다.

완화책: 추가 전용 로그, 출처(provenance), 쓸 수 없는 검증자. 이미 레슨 13에서 다뤘다.

### STRATUS — 실패 탐지를 위한 특화 에이전트

STRATUS (NeurIPS 2025)는 다음을 배포할 때 완화 성공이 1.5배 개선됨을 보고한다.

- **탐지 에이전트.** 증상 패턴(높은 불일치, 재시도 급증, 정확도 표류)을 감시한다.
- **진단 에이전트.** 증상이 주어지면, MAST 분류 체계에서 가능성 있는 근본 원인을 추론한다.
- **검증 에이전트.** 완화책이 적용된 후, 증상이 해소되는지 검사한다.

이것은 에이전트 시스템에 적용된 SRE 스타일 사고 대응이다. 세 역할 모두 특화된 프롬프트를 가진 LLM 에이전트일 수 있다.

### 실패 모드 감사

2026년 모범 사례는 연간(또는 주요 릴리스마다) 실패 모드 감사다.

1. **추적 샘플.** 약 1000개의 실제 실행 추적을 수집하라.
2. **분류.** 각 추적의 실패를 MAST + 집단사고 범주에 매핑하라.
3. **범주별 실패율 계산.** 당신의 시스템에서 어떤 범주가 지배적인가?
4. **완화책 순위 매기기.** 어떤 수정이 가장 많은 실패를 제거할까?
5. **2-3개의 완화책 선택.** 구현하라. 다음 분기에 재감사하라.

규율이 특정 선택보다 더 중요하다. 감사 없이는, 실패가 잡음에 섞여 체계적으로 다뤄지지 않는다.

### 시스템이 조용히 실패할 때

가장 위험한 실패 범주는 조용한 정확성 실패다. 시끄럽게 실패하는(충돌, 예외, 경보) 시스템은 모니터링할 수 있다. 그럴듯하지만 틀린 출력을 만드는 시스템은 예외 로그로 탐지할 수 없다. 이것이 검증 격차가 개수로는 21.30%에 불과하더라도 실패당 가장 비싼 범주인 이유다.

다음에 투자하라:
- 샘플 기반 인간 검토.
- 골든 데이터셋(golden-dataset) 회귀 테스트.
- 중요한 출력에 대한 에이전트 간 교차 검사.

### 실패 대 느린 실패

일부 실패는 즉각적이고, 일부는 느리다. 즉각적 실패(타임아웃, 스키마 불일치, 인증 오류)는 탐지가 저렴하다. 느린 실패(메모리 오염, 단일 문화 표류, 역할 모호성)는 탐지와 예방이 비싸다.

2026년 엔지니어링 수: 느린 실패 대리 지표를 계측하여 표류가 가시적 오류가 되기 전에 잡아라. 합의율, 재시도율, 출력 길이 분포, 연속된 에이전트 버전 간 편집 거리는 모두 유용한 대리 지표다.

## 직접 만들기 (Build It)

`code/main.py`는 다음을 구현한다.

- `FailureTaxonomy` — 시뮬레이션된 사고를 MAST + 집단사고 범주로 분류한다.
- `CircuitBreaker` — 고전적 패턴. 오류율이 임계값을 초과하면 열린다.
- `RetryStormSimulator` — 연쇄 실패를 보여준다. 회로 차단기를 켜고/끈다.
- `DetectionAgent` — 스크립트화된 STRATUS 스타일 증상 매처.

실행:

```
python3 code/main.py
```

예상 출력:
- 회로 차단기 없는 재시도 폭풍: 재고 오류가 폭발한다(시뮬레이션).
- 회로 차단기 있음: 임계값에서 상한. 강등 모드 응답이 제공된다.
- 탐지 에이전트가 패턴에 플래그를 달고 MAST 범주의 이름을 댄다.

## 라이브러리로 써보기 (Use It)

`outputs/skill-mast-auditor.md`는 다중 에이전트 시스템에 MAST 스타일 실패 모드 감사를 실행한다. 추적 → 분류 → 완화책 순위 매기기.

## 산출물 (Ship It)

프로덕션의 실패 모드 규율:

- **분기마다 MAST 감사.** 연간이 아니다. 시스템이 성장하면서 범주가 바뀐다.
- **어디서나 회로 차단기.** 어떤 의존 서비스로든 나가는 각 호출에. 기본 개방 임계값은 5-10% 오류율.
- **골든 데이터셋.** 작고, 고품질이며, 수작업으로 감사된 것. 매주 그것에 대해 회귀 테스트하라.
- **STRATUS 삼인조.** 프로덕션을 모니터링하는 탐지 + 진단 + 검증 에이전트. 탐지 에이전트만으로 시작하라. 증상이 시끄러울 때 진단을 추가하라.
- **실패 예산.** 범주별 실패율에 대한 명시적 SLO. 예산 초과는 출시 중단 대화를 유발한다.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 회로 차단기가 재시도 폭풍에 상한을 두는지 확인하라. 실패 임계값을 바꿔 트레이드오프(trade-off)를 관찰하라.
2. **느린 실패 대리 지표**를 구현하라: 3개의 병렬 에이전트에 걸친 합의율. 그것이 급격히 떨어지면 경보를 유발하라. 에이전트 출력을 점진적으로 상관시켜 단일 문화 표류를 시뮬레이션하라.
3. Cemri et al. (arXiv:2503.13657)을 읽어라. 그들의 7개 MAS 시스템 중 하나를 골라 그 상위 3개 실패 범주를 매핑하라. 이것들이 MAST가 예측하는 것과 어떻게 비교되는가?
4. 집단사고 논문(arXiv:2508.05687)을 읽어라. 다섯 패턴 중 프로덕션에서 탐지하기 가장 어려운 것을 식별하라. 대리 지표를 제안하라.
5. 당신이 아는 특정 다중 에이전트 시스템에 대한 STRATUS 스타일 탐지-진단-검증 삼인조를 설계하라. 탐지는 어떤 증상을 감시하는가? 진단은 어떤 완화책을 권하는가? 검증은 그것들이 작동하는지 어떻게 확인하는가?

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| MAST | "2026년 분류 체계" | Cemri 2025. 3개의 근본 범주 + 14개의 실패 하위 유형. |
| 명세 문제(Specification Problem) | "역할 모호성" | 과제나 역할이 미정의됨. 에이전트들이 무엇을 할지 모름. |
| 협응 실패(Coordination Failure) | "상태 표류" | 에이전트 간 통신 또는 동기화 붕괴. |
| 검증 격차(Verification Gap) | "아무도 검사하지 않음" | 독립적 검증 없이 받아들여진 출력. |
| 집단사고 계열(Groupthink family) | "동질성 실패" | 단일 문화, 동조, 결함 있는 ToM, 혼합 동기, 연쇄. |
| 단일 문화 붕괴(Monoculture collapse) | "같은 모델, 같은 환각" | 공유 기반 모델 또는 학습 데이터로부터의 상관된 오류. |
| 재시도 폭풍(Retry storm) | "연쇄 오류 증폭" | 한 실패가 재시도를 유발하고 그것이 하류 부하를 증폭. |
| 회로 차단기(Circuit breaker) | "오류율에 빠르게 실패" | 오류율이 임계값을 초과하면 열림. 기본값으로 단락. |
| STRATUS | "사고 대응 삼인조" | 탐지 + 진단 + 검증 에이전트. 완화 성공 1.5배. |
| 메모리 오염(Memory poisoning) | "환각이 전파됨" | 공유 메모리 사실이 오염됨. 하류 에이전트들이 오염물에 대해 추론. |
