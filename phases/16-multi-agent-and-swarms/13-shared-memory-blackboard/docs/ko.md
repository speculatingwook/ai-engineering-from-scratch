# 공유 메모리와 블랙보드 패턴 (Shared Memory and Blackboard Patterns)

> 2026년 멀티 에이전트 시스템에는 두 접근법이 공존한다. **메시지 풀(message pool)**(AutoGen GroupChat나 MetaGPT처럼 모두가 모두의 메시지를 본다)과 **구독 기반 블랙보드(blackboard with subscription)**(Context-Aware MCP나 Matrix 프레임워크처럼 에이전트가 관련 이벤트를 구독한다)다. 둘 다 멀티 에이전트 시스템에서 유일하게 상태를 가진 부분이며, 그래서 흥미로운 버그가 사는 곳이다. 대표적 실패 모드는 **메모리 오염(memory poisoning)**이다. 한 에이전트(agent)가 "사실"을 환각(hallucinate)하면, 다른 에이전트들이 그것을 검증된 것으로 취급하고, 정확도가 즉각적인 충돌보다 훨씬 디버깅하기 어려운 방식으로 서서히 무너진다. 이 레슨은 두 구조를 stdlib로 만들고, 오염 공격을 주입하고, 프로덕션(production)에서 실제로 통하는 세 가지 완화책을 보여준다.

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`)
**Prerequisites:** Phase 16 · 04 (Primitive Model), Phase 16 · 09 (Parallel Swarm Networks)
**Time:** ~75분

## 문제 (Problem)

멀티 에이전트 시스템은 에이전트들이 사실을 공유할 장소가 필요하다. 글자 그대로의 한 선택지는 "모든 것을 메시지로 전달"하는 것이지만, 이는 추가 복사를 동반해 공유 상태를 재발명한다. 또 다른 선택지는 "모두에게 전역 로그를 준다"는 것이지만, 전역 로그는 무한정 커지고 쉽게 오염된다. 세 번째는 "에이전트마다 뷰(view)를 프로젝션한다"는 것이지만, 확장 가능하되 스키마가 무겁다.

에이전트 중 하나가 환각을 일으켜 그 환각을 공유 상태에 쓰면, 그 상태를 읽는 모든 하류(downstream) 에이전트가 그 환각을 사실로 받아들인다. 사람이 알아챌 무렵이면 추론 사슬은 다섯 단계 깊이에 들어가 있고, 근본 원인은 애초에 세 번째로 쓰인 메시지다. 멀티 에이전트의 정확도 붕괴를 디버깅하는 것은 충돌을 디버깅하는 것보다 어렵다.

이것이 메모리 오염이다. 이는 MAST 분류 체계(Cemri et al., arXiv:2503.13657)에서 두 번째로 많이 문서화된 실패 계열이며 구조적이다. 출처(provenance)와 쓸 수 없는 검증자가 없는 어떤 공유 메모리 설계든 결국 이를 드러낸다.

## 개념 (Concept)

### 두 가지 주요 토폴로지

**전체 메시지 풀(Full message pool).** 모든 에이전트가 모든 메시지를 읽는다. AutoGen GroupChat와 MetaGPT가 이를 쓴다. 단순하고, 투명하고, 검사 가능하지만, 각 에이전트의 컨텍스트(context)가 다른 에이전트들의 작업으로 채워지기 때문에 ~10개 에이전트를 넘어서면 확장되지 않는다.

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**구독 기반 블랙보드(Blackboard with subscription).** 에이전트가 토픽에 대한 관심을 선언하면, 기반(substrate)이 관련 메시지만 라우팅한다. CA-MCP(arXiv:2601.11595)와 Matrix 탈중앙화 프레임워크(arXiv:2511.21686)가 이를 쓴다. 더 멀리 확장되지만, 구독을 의미 있게 만들려면 사전 스키마 설계가 필요하다.

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 각각이 이길 때

- **전체 풀**은 에이전트가 적고(< 10), 이질적이며, 대화가 짧은 지평일 때 이긴다. 모두가 모든 것을 볼 때 누가 무엇을 말했는지 추론하기는 자명하다.
- **블랙보드**는 에이전트가 많고, 역할은 동질적이되 인스턴스가 많으며(스웜(swarm)), 대화가 장기 실행일 때 이긴다. 라우팅이 토큰 비용과 컨텍스트 오염을 절약한다.

프로덕션 시스템은 종종 둘을 섞는다. 위쪽(계획 층)에 작은 전체 풀, 아래쪽(워커 층)에 블랙보드.

### 한 시나리오로 보는 메모리 오염

세 에이전트가 연구 작업을 한다. 에이전트 A는 검색(retrieval) 에이전트다. 에이전트 B는 요약자다. 에이전트 C는 분석가다.

1. A가 페이지를 가져와 공유 상태에 메시지를 쓴다: "이 연구는 42% 정확도 향상을 보고한다."
2. 가져온 페이지는 실제로 "4.2% 향상"이라고 했다. A가 소수점을 환각했다.
3. B가 공유 상태를 읽고 쓴다: "큰 42% 정확도 향상이 보고됨 (출처: A)."
4. C가 공유 상태를 읽고 쓴다: "도입 권장 — 42% 향상은 혁신적이다."
5. 최종 보고서는 존재한 적 없는 42%라는 수치를 인용한다.

어떤 에이전트도 충돌하지 않았다. 어떤 테스트도 실패하지 않았다. 시스템은 "작동했다." 환각은 공유 상태를 통해 한 에이전트의 컨텍스트에서 모든 하류 에이전트의 추론으로 건너갔다.

### 왜 이것이 구조적인가

공유 상태가 없다면 에이전트 A의 환각은 A의 컨텍스트에 머문다. 하류 에이전트들은 다시 가져오거나 다시 유도하여 오류를 잡아낼 수 있다. 순진한 공유 상태가 있으면 A의 컨텍스트가 모두의 컨텍스트가 되고, 환각이 사실로 세탁된다.

문제는 공유 상태 그 자체가 아니다. 문제는 **출처 없이, 독립적 검증자 없이** 존재하는 공유 상태다. 세 가지 완화책이 이를 다룬다.

1. **모든 쓰기에 출처를 기록한다.** 공유 상태의 모든 항목은 누가, 언제, 어떤 프롬프트(prompt) 하에서 썼는지, 그리고 (해당된다면) 에이전트가 인용한 출처가 무엇인지 기록한다. 하류 에이전트는 출처에 맞춰진 회의(skepticism)로 읽는다.
2. **쓰기를 버전화한다. 추가 전용(append-only)으로 다룬다.** 수정은 기존 것을 대체하는 새 항목이지, 제자리 갱신이 아니다. 감사 추적(audit trail)이 보존된다.
3. **공유 상태에 쓸 수 없는 에이전트를 최소 하나 둔다.** 읽기 전용 검증자 에이전트가 항목을 표본 추출하고, 출처를 다시 가져오고, 불일치를 표시한다. 풀에 쓸 수 없기 때문에 풀에 의해 오염될 수 없다.

### 블랙보드의 선례 (Hayes-Roth, 1985)

블랙보드 패턴은 LLM 에이전트보다 40년 앞선다. Hayes-Roth(1985, "A Blackboard Architecture for Control")는 전역 블랙보드를 관찰하고, 부분 해법을 기여하고, 다른 소스를 촉발하는 전문가 지식 소스(Knowledge Source)를 기술했다. 2026년 블랙보드(CA-MCP, Matrix)는 LLM 에이전트를 지식 소스로, JSON 블롭(blob)을 부분 해법으로 둔 동일한 패턴이다. 과거 문헌은 현대 시스템이 재발견하는 쓰기 경합(write contention), 기회주의적 제어, 일관성 해법을 이미 문서화해 두었다.

### 프로젝션 vs 전체 뷰

순수 블랙보드는 모든 구독자에게 같은 프로젝션(토픽 범위)을 준다. 더 공격적인 설계는 **에이전트별 프로젝션(per-agent projection)**이다. 각 에이전트가 자신의 역할에 맞춰진 뷰를 받는다. LangGraph의 상태 리듀서(state reducer)가 정전(canonical) 2026 구현이다. 리듀서 함수가 전역 상태를 역할별 슬라이스로 접는다.

에이전트별 프로젝션은 더 멀리 확장되지만 스키마가 필요하다. 스키마가 없으면 모든 에이전트의 프롬프트에서 임시방편 프로젝션을 다시 만들게 된다.

### 쓰기 경합 패턴

여러 에이전트가 동시에 쓰는 것은 LLM 문제만이 아니라 동시성(concurrency) 문제다. 세 가지 패턴이 통한다.

- **순차 작성자(단일 생산자).** 모든 쓰기가 직렬화하는 하나의 코디네이터(coordinator) 에이전트를 거친다. 단순하지만 병목이다.
- **버전화를 동반한 낙관적 동시성(optimistic concurrency).** 각 항목에 버전이 있다. 작성자는 버전 불일치 시 실패하고 재시도한다. 고전적 데이터베이스 기법.
- **토픽 분할(Topic partitioning).** 서로 다른 에이전트가 서로 다른 토픽을 소유한다. 토픽 간 경합이 없다. 설계된 분할 경계가 필요하다.

대부분의 2026 프레임워크는 순차 작성자를 기본값으로 한다. LLM 호출이 충분히 느려서 경합이 드물고 병목이 해롭지 않기 때문이다.

### 쓸 수 없는 검증자

가장 하중을 견디는 완화책은 읽기 전용 검증자다. 구현 규칙은 다음과 같다.

- 검증자는 팀과 상태를 공유한다(블랙보드나 풀을 읽는다).
- 검증자는 공유 상태에 대한 쓰기 핸들이 없다. 별도의 검증 채널에만 쓸 수 있다.
- 검증자는 쓰기에서 인용된 출처를 독립적으로 가져온다. 불일치를 표시한다.
- 검증자 자신의 출력은 사람이나 별도의 결정 에이전트로 라우팅되며, 절대 풀로 되먹임되지 않는다.

이 분리가 없으면 검증자의 출력이 풀의 새 항목이 되고, 그러면 오염된 풀이 검증자를 오염시키며, 그 검증마저 오염된다.

## 직접 만들기 (Build It)

`code/main.py`는 두 토폴로지를 stdlib 파이썬으로 구현하며, 장난감 오염 공격과 세 가지 완화책을 더한다.

- `MessagePool` — 전체 읽기가 가능한 스레드 안전(thread-safe) 추가 전용 로그.
- `Blackboard` — 에이전트별 구독을 동반한 토픽 키 기반 pub/sub.
- `ProvenanceEntry` — 모든 쓰기가 (writer, timestamp, prompt_hash, source_uri)를 기록한다.
- `PoisoningScenario` — 에이전트 A가 소수점을 환각하는 세 에이전트 연구 작업을 실행한다. 최종 보고서를 출력한다.
- `Verifier` — 출처를 다시 가져와 불일치를 표시하는 읽기 전용 에이전트. 검증자가 있는 상태로 동일한 시나리오를 실행한다.

실행:

```
python3 code/main.py
```

예상 출력:
- 실행 1 (검증자 없음): 환각된 42%가 최종 보고서로 전파된다.
- 실행 2 (검증자 있음): 검증자가 불일치를 표시하고, 풀이 "flagged"로 라벨링되며, 최종 보고서에 철회(retraction)가 포함된다.

## 라이브러리로 써보기 (Use It)

`outputs/skill-memory-auditor.md`는 임의의 멀티 에이전트 시스템의 공유 메모리 설계를 출처, 버전화, 검증자 분리 관점에서 감사하는 스킬이다. 새 멀티 에이전트 아키텍처를 프로덕션에 올리기 전에 실행하라.

## 산출물 (Ship It)

모든 공유 메모리 설계에 대해:

- 모든 쓰기에 출처를 기록하라: `(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`.
- 로그를 추가 전용으로 만들어라. 수정은 대체된 항목을 참조하는 새 항목이다.
- 독립적 출처 접근을 가진 읽기 전용 검증자 에이전트를 최소 하나 배포하라.
- 검증자 출력을 공유 풀로 되돌리지 말고 별도 채널로 라우팅하라.
- 쓰기 중 대체(supersession)의 비율을 기록하라. 비율이 오르는 것은 환각 패턴의 초기 증거다.

## 연습 문제 (Exercises)

1. `code/main.py`를 실행하라. 실행 1이 환각을 전파하고 실행 2가 이를 잡아내는지 확인하라.
2. 두 번째 환각을 추가하라. 에이전트 B가 데이터셋(dataset) 크기를 지어낸다. 검증자는 어느 쪽으로도 수동 튜닝되지 않고 둘 다 잡아내야 한다.
3. 전체 풀을 토픽 분할(`prices`, `summaries`, `analyses`)을 동반한 블랙보드로 바꿔라. 토픽 분할이 어떤 오염 시나리오를 더 어렵게 만들고, 어떤 시나리오에는 도움이 안 되는가?
4. Hayes-Roth(1985, "A Blackboard Architecture for Control")를 읽어라. 이 레슨에서 논의하지 않았지만 2026 시스템이 이득을 볼 두 가지 제어 패턴을 논문에서 식별하라.
5. CA-MCP(arXiv:2601.11595)를 읽어라. 그것의 Shared Context Store를 `code/main.py`의 `MessagePool` 또는 `Blackboard` 클래스에 매핑하라. CA-MCP가 그 위에 추가하는 프리미티브는 무엇인가?

## 핵심 용어 (Key Terms)

| 용어 | 사람들이 말하는 것 | 실제 의미 |
|------|----------------|------------------------|
| 메시지 풀(Message pool) | "공유 채팅 이력" | 모든 에이전트가 읽는 추가 전용 로그. 완전한 투명성, 부실한 확장성. |
| 블랙보드(Blackboard) | "공유 작업 공간" | 토픽 키 기반 pub/sub. 에이전트가 관련 토픽을 구독한다. 더 멀리 확장된다. |
| 출처(Provenance) | "누가 무엇을 썼나" | 각 쓰기에 대한 메타데이터: 작성자, 타임스탬프, 프롬프트, 출처. |
| 메모리 오염(Memory poisoning) | "환각이 퍼짐" | 한 에이전트의 오류가 공유 상태에 들어가고, 하류 에이전트가 이를 사실로 받아들인다. |
| 추가 전용(Append-only) | "제자리 갱신 없음" | 수정은 대체하는 새 항목이다. 감사 추적을 보존한다. |
| 쓸 수 없는 검증자(Unwritable verifier) | "독립적 감사자" | 출처를 다시 가져와 불일치를 표시하는 읽기 전용 에이전트. |
| 프로젝션(Projection) | "범위 제한 뷰" | 전역 상태로부터 계산된 에이전트별 뷰. LangGraph 리듀서가 정전적 사례다. |
| 지식 소스(Knowledge Source) | "전문가 에이전트" | 블랙보드 참여자를 가리키는 Hayes-Roth의 1985년 용어. |
